4.6 Documentation

← Topic 4

Requirements documentation is the activity of recording analysed requirements in a form that is understandable, reviewable, and maintainable. The primary deliverable is usually the Software Requirements Specification (SRS), but documentation also includes models, glossaries, and supporting artefacts.

Why document requirements?

  • Documentation creates a shared contract between customer and development team.
  • It provides the baseline for design, implementation, testing, and acceptance.
  • It supports traceability — each requirement can be linked to its source and to test cases.
  • It enables impact analysis when changes are proposed during the project.
  • It preserves knowledge when team members leave or when the system enters maintenance.

Forms of requirements documentation

  • Natural language (SRS): The main written document describing functional and non-functional requirements in prose (see §4.14).
  • Graphical models: DFDs, ERDs, use-case diagrams, and state diagrams that complement the text (see §4.11, §4.12).
  • Structured notations: Decision tables for complex conditional logic (see §4.13).
  • User stories / use cases: Scenario-based descriptions favoured in Agile methods.
  • Glossary and data dictionary: Definitions of domain terms and data elements to prevent ambiguity.

Documentation standards and guidelines

  • Each requirement should have a unique identifier for traceability.
  • Requirements should be stated at a consistent level of abstraction — neither too vague nor too detailed.
  • Each statement should express a single requirement; compound sentences should be split.
  • Active voice and defined terminology improve clarity.
  • IEEE 830 / IEEE 29148 provide standard SRS structure (see §4.17).

Document control

Requirements documents must be version-controlled and subject to formal review. Only approved versions become the requirements baseline. Changes after baseline approval follow the project's change control process, managed as part of requirements management in §4.7 and SPM scope control in Topic 3 §3.9.

Exam question — Requirements documentation

Q: Why must requirements be documented? What forms of documentation are used?

Model answer points:

  • Explain purpose: shared contract, design/test baseline, traceability, change impact, knowledge preservation.
  • List forms: SRS (natural language), graphical models (DFD/ERD), decision tables, use cases, glossary.
  • State documentation rules: unique ID, consistent abstraction, one req per statement, defined terms.
  • Mention IEEE SRS standard structure and document version control/baseline.
  • Link to review, management, and SPM change control.
