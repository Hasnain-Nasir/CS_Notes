4.10 System Modelling / Information Modeling

← Topic 4

System modelling (also called information modelling) uses graphical and structured notations to represent requirements visually. Models complement the written SRS by making data flows, data structures, and system behaviour easier to understand, review, and validate.

Why model during requirements engineering?

  • Models expose gaps, inconsistencies, and ambiguities that prose alone may hide.
  • They provide a shared visual language between analysts, developers, and stakeholders.
  • They serve as a bridge from requirements to design — many design diagrams evolve directly from RE models.
  • They support validation: stakeholders can confirm that a diagram matches their understanding of the system.

Common modelling techniques in RE

  • Data Flow Diagrams (DFD): Show how data moves through the system, which processes transform it, and where it is stored. Covered in §4.11.
  • Entity Relationship Diagrams (ERD): Show the data entities, their attributes, and the relationships between them. Covered in §4.12.
  • Decision tables: Tabulate complex conditional logic for unambiguous specification. Covered in §4.13.
  • Use case diagrams: Show actors and the use cases (services) they interact with — common in object-oriented RE.
  • State transition diagrams: Show how the system or an entity changes state in response to events.

Information modelling concepts

  • Data flow modelling focuses on functional processing — what happens to data as it moves through the system (DFD).
  • Data structure modelling focuses on what data exists and how entities relate (ERD).
  • Both views must be consistent: every data store in a DFD should correspond to entities in the ERD, and every process should be supported by the documented functional requirements.

Modelling guidelines

  • Keep models at the requirements level — avoid premature design detail such as specific algorithms or UI widgets.
  • Use standard notation so all team members interpret symbols consistently.
  • Number and cross-reference diagrams in the SRS document.
  • Review models with stakeholders alongside the written requirements.
Exam question — System modelling

Q: Why is system modelling used in requirements engineering? Name three modelling techniques.

Model answer points:

  • Explain purpose: expose gaps/inconsistencies, shared visual language, bridge to design, support validation.
  • Name DFD (data flow), ERD (data structure), decision tables, use cases, state diagrams.
  • Distinguish data flow modelling from data structure modelling.
  • State consistency rule: DFD data stores align with ERD entities.
  • Give modelling guidelines: requirements-level, standard notation, cross-reference, stakeholder review.
