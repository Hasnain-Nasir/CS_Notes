Key takeaways — exam revision

← Topic 4
  • Requirements Engineering (RE): Systematic process of discovering, documenting, analysing, validating, and managing requirements. IEEE 729: condition/capability needed by user or imposed by contract/standard.
  • RE process (5 phases): Feasibility (technical, operational, economic, legal, schedule) → Elicitation (interviews, surveys, focus groups, observation, prototyping) → Specification (functional, non-functional, constraints, acceptance criteria; DFD/ERD/FDD) → V&V (verification = correct; validation = meets needs; iterative) → Management (change control, traceability, version control). Tools: workshops, use cases, mind mapping, role-playing, prototyping.
  • Feasibility types: Technical, economic, operational, schedule (+ legal, social). Output: feasibility report → go/no-go.
  • Elicitation techniques: Interviews, brainstorming, FAST, QFD, use cases, observation, questionnaires, document analysis, prototyping. Steps: identify stakeholders → understand domain → select techniques → conduct → record → validate.
  • Elicitation challenges: Unknown needs, conflicts, changing requirements, terminology gaps, implicit assumptions, domain complexity. Strategies: multiple techniques, prototyping, glossary, prioritisation, change control.
  • Analysis objectives: Resolve conflicts, find gaps/redundancies, clarify ambiguity, prioritise (MoSCoW), classify, model. Outputs: refined list, models, traceability matrix.
  • Documentation forms: SRS (natural language), DFD/ERD models, decision tables, use cases, glossary. Rules: unique ID, one req per statement, version control, baseline.
  • Review vs management: Review = validation before design (inspections, walkthroughs, prototyping). Management = version control, change control (CCB), traceability, impact analysis throughout SDLC.
  • Requirements classification (IEEE 729): Functional (what — input/operation/output; functional specification), non-functional (how — execution qualities: security/usability; evolution qualities: maintainability/scalability; types: interface, performance, operating, life cycle, economic), domain (industry-specific — HIPAA, GAAP; from domain model). Also: user/system/business/regulatory/interface/design.
  • System modelling: DFD (data flow), ERD (data structure), decision tables, use cases. DFD stores must align with ERD entities.
  • DFD: Graphical tool for data movement; 4 components (process, data flow, data store, external entity). Levels: 0 (context), 1, 2+. Logical (what — no implementation) vs physical (how — hardware/DB). Rules: can flow entity→process, process→store; cannot flow entity→entity, store→store. Railway example: Search/Book/Cancel/Payment; Level 2 decomposes Book Ticket.
  • ERD: Entity, attribute, relationship, key. Cardinality: 1:1, 1:N, M:N. Strong vs weak entity. Steps: identify entities → attributes/keys → relationships → draw → review.
  • Decision tables: Condition stubs, action stubs, entries (Y/N/—, X). Cause-effect. Advantages: unambiguous, reveals gaps/redundancy, basis for test cases.
  • SRS: Contract between customer & developer; layout of software behaviour. Sections: Introduction (purpose, scope, overview) → General description → Functional (I/O, ranked) → Interface → Performance (static/dynamic) → Design constraints → Non-functional → Schedule/budget → Appendices → Goals of implementation. Uses: dev, testing, PM, maintenance, contract. Importance: clarity, planning, QA, change control, risk mitigation. Qualities: complete, correct, consistent, verifiable, traceable, testable, unambiguous.
  • SRS quality (12): Correct, complete, consistent, unambiguous, ranked, modifiable, verifiable, traceable, design-independent, testable, understandable, right abstraction level.
  • IEEE 830 SRS structure: §1 Introduction (purpose, scope, definitions, references, overview) → §2 Overall description (perspective, functions, users, constraints, assumptions) → §3 Specific requirements (external interfaces, functions, performance, database, design constraints, system attributes: reliability/availability/security/maintainability/portability) → Appendices. 8 qualities: correct, unambiguous, complete, consistent, ranked, verifiable, modifiable, traceable. Black box spec — no design/project details in SRS.
  • IEEE 29148: Full RE process; stakeholder/system/software req levels; each req needs ID, statement, rationale, source, priority, verification method (inspection/analysis/demo/test). Bidirectional traceability. IEEE 729: requirement glossary definition.
  • Software specification: Formal documented expression of requirements (SRS + models). Requirements → specification → design. Methods: natural language, structured, graphical, formal, executable.
