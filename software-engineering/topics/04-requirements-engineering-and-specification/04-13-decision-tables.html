4.13 Decision Tables

← Topic 4

A decision table is a tabular representation of complex conditional logic. It lists all combinations of conditions and the corresponding actions the system should take. Decision tables are used during requirements analysis and specification when prose descriptions of business rules would be ambiguous or error-prone (see §4.5).

Structure of a decision table

  • Condition stubs: Rows listing the conditions (inputs) that affect the decision, such as "Account balance sufficient?" or "Customer type = Premium?"
  • Action stubs: Rows listing the possible actions (outputs) the system can take, such as "Approve transaction" or "Apply discount."
  • Condition entries: Columns showing Y (yes/true), N (no/false), or — (don't care/not applicable) for each condition under each rule column.
  • Action entries: Cells indicating which actions are executed (marked X) for each rule column.

Cause-effect table

A cause-effect table is closely related to a decision table. Causes are input conditions; effects are output actions. The table maps every meaningful combination of causes to the resulting effects, making it useful for deriving test cases as well as specifying requirements.

Example: loan approval

Consider conditions: Credit score ≥ 700, Income ≥ threshold, Existing debt low. Actions: Approve loan, Reject loan, Request manual review. Each column represents one rule; the analyst fills in which actions fire for each condition combination.

Advantages of decision tables

  • They make complex conditional logic explicit and unambiguous — no interpretation needed.
  • They reveal missing rules: if a condition combination has no action, a gap exists in the requirements.
  • They reveal redundant or contradictory rules when two columns specify conflicting actions for the same conditions.
  • They serve directly as a basis for test case design — each column is one test scenario.
  • They are compact compared to lengthy nested if–else prose in an SRS.

Limitations

  • Tables become very large when there are many conditions (2n combinations for n binary conditions).
  • They are best suited to discrete, boolean conditions rather than continuous numeric ranges.
  • Stakeholders unfamiliar with the notation may need explanation during review.
Exam question — Decision tables

Q: What is a decision table? Explain its components and advantages.

Model answer points:

  • Define decision table as tabular representation of conditional logic.
  • Name components: condition stubs, action stubs, condition entries (Y/N/—), action entries (X).
  • Explain cause-effect relationship and link to test case derivation.
  • List advantages: unambiguous, reveals gaps/redundancy, compact, testable.
  • Mention limitation: exponential growth with many conditions.
