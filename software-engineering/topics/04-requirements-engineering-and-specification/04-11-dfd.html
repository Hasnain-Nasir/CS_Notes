4.11 Data Flow Diagrams (DFD)

← Topic 4

A Data Flow Diagram (DFD) is a graphical representation of the flow of data through a system. It shows external entities, processes that transform data, data stores, and the data flows connecting them. DFDs are a core system modelling tool in requirements engineering (see §4.10).

Four basic DFD components

  • Process (bubble): An activity that transforms incoming data into outgoing data. Named with a verb phrase (e.g. "Validate Order").
  • Data flow (arrow): Movement of data between processes, stores, and external entities. Named with a noun phrase describing the data (e.g. "Order Details").
  • Data store (open rectangle): A repository where data is held for later use (e.g. "Customer File").
  • External entity (square): A source or destination of data outside the system boundary (e.g. "Customer," "Bank").

Levels of DFD

  • Context diagram (Level 0): Shows the entire system as a single process with external entities and major data flows. No internal detail.
  • Level 1 DFD: Explodes the context process into major sub-processes, showing internal data flows and stores.
  • Level 2 and below: Each Level 1 process can be further decomposed until each process is simple enough to describe in the SRS or map to a module.

Logical vs physical DFD

  • Logical DFD: Shows what the system does regardless of how it is implemented. Used during requirements analysis to capture business processes.
  • Physical DFD: Shows how the system is actually implemented — specific files, devices, and human roles. Used later in design.

DFD drawing rules

  • Each process must have at least one input and one output.
  • Data flows must be labelled; unlabelled arrows are ambiguous.
  • A data store is accessed by at least one process (no "orphan" stores).
  • External entities cannot communicate directly with each other — all interaction goes through the system.
  • Processes are numbered hierarchically (1, 1.1, 1.1.1) to show decomposition levels.
  • Avoid control flow (decision diamonds) — DFDs show data flow, not control logic.

Railway reservation example

A railway reservation system illustrates DFD levelling:

  • Level 0: A single "Railway Reservation System" process interacts with Passenger, Payment Gateway, and Railway Admin external entities.
  • Level 1: The system is decomposed into sub-processes such as Search Trains, Book Ticket, Process Payment, and Generate Ticket, with data stores for Reservation Database and Train Schedule.
Exam question — DFD components and levels

Q: Explain DFD components and levels. What are the main drawing rules?

Model answer points:

  • Name four components: process, data flow, data store, external entity.
  • Describe Level 0 (context), Level 1, and further decomposition.
  • Distinguish logical DFD (what) from physical DFD (how).
  • State rules: labelled flows, no direct entity-to-entity, hierarchical numbering, no control flow.
  • Sketch or describe railway reservation Level 0 and Level 1 example.
