4.18 Software Specification (general)

← Topic 4

Software specification is the broader discipline of describing what a software system should do and the constraints under which it must operate. The SRS is the primary specification document, but specification also encompasses models, interface specs, and standards that together define the product before design and coding begin.

Specification vs requirements vs design

  • Requirements express stakeholder needs — often informal initially, then refined.
  • Specification is the formal, documented expression of those requirements in a structured, reviewable form (the SRS and its supporting models).
  • Design describes how the specification will be realised in code and architecture — it follows specification, not replaces it.

Types of software specifications

  • Requirements specification (SRS): Complete description of functional and non-functional requirements (§4.14).
  • Interface specification: Detailed description of APIs, protocols, and data formats at system boundaries.
  • Design specification: Produced in the design phase — architecture, module structure, database schema. Not part of RE but derived from the SRS.
  • Test specification: Test plans and cases derived from requirements for verification.

Role in the SDLC and SPM

Specification is the key output of the requirements phase in the SDLC. It enables the project manager to baseline scope for estimation and control (see Topic 3). Without a clear specification, the team cannot reliably estimate effort, plan sprints, or define acceptance criteria.

Specification methods

  • Natural language: Prose in the SRS — flexible but prone to ambiguity.
  • Structured natural language: Templates with fixed fields (Shall-statements with IDs).
  • Graphical models: DFD, ERD, UML diagrams (see §4.10).
  • Formal methods: Mathematical notations (Z, VDM) for safety-critical systems where ambiguity is unacceptable.
  • Executable specification: Prototypes or domain-specific languages that can be run to validate behaviour.

Benefits of thorough specification

  • Reduces rework by catching misunderstandings before coding.
  • Provides a legally and contractually meaningful agreement with the customer.
  • Enables parallel work — designers, testers, and technical writers can start from the spec while coding proceeds.
  • Supports maintenance and enhancement by documenting the intended behaviour of the delivered system.
Exam question — Software specification

Q: What is software specification? How does it relate to requirements and design?

Model answer points:

  • Define specification as formal documented description of what system should do and its constraints.
  • Distinguish requirements (needs) → specification (formal SRS/models) → design (how to build).
  • List types: requirements spec (SRS), interface spec, design spec, test spec.
  • Link to SDLC requirements phase output and SPM scope baseline.
  • Name methods: natural language, structured, graphical, formal, executable; state benefits.
