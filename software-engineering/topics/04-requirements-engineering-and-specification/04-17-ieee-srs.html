4.17 IEEE Standards for SRS

← Topic 4

The IEEE has published standards that define recommended structure and content for Software Requirements Specifications. The most widely referenced are IEEE Std 830-1998 (superseded but still cited in exams) and IEEE Std 29148-2011, which consolidates and updates earlier RE standards.

IEEE 830 SRS recommended structure

IEEE 830 defines the following major sections for an SRS:

  1. Introduction — purpose, scope, definitions, references, overview.
  2. Overall description — product perspective, functions, user characteristics, constraints, assumptions and dependencies.
  3. Specific requirements — the bulk of the document:
    • External interface requirements (user, hardware, software, communication).
    • Functional requirements.
    • Performance requirements.
    • Design constraints.
    • Software system attributes (reliability, availability, security, maintainability, portability).
    • Other relevant requirements (database, internationalisation, legal).
  4. Appendices — sample I/O, glossary, analysis models.
  5. Index — optional cross-reference index.

IEEE 29148 key concepts

IEEE 29148 (2011) extends IEEE 830 and covers the full RE process, not only the SRS document:

  • Defines terms for requirements, stakeholder, verification, and validation consistent with ISO/IEC/IEEE 15288 (systems engineering).
  • Specifies content for three types of requirements information: stakeholder requirements (what stakeholders need), system requirements (what the system shall do), and software requirements (detailed SW-level specs).
  • Requires each requirement to include: a unique ID, a statement, rationale, source, priority, and verification method (inspection, analysis, demonstration, test).
  • Emphasises bidirectional traceability between stakeholder requirements, system requirements, design, code, and tests.

IEEE 729 — vocabulary

IEEE Std 729-1983 provides the standard glossary for software engineering terminology, including the definition of "requirement" used throughout this topic (see §4.1 and §4.9).

Why follow IEEE standards?

  • Provides a recognised, complete template so no important section is omitted.
  • Facilitates review and contractual agreement between customer and supplier.
  • Supports certification and compliance in regulated industries.
  • Aligns with international systems and software engineering standards (ISO/IEC 12207, 15288).
Exam question — IEEE SRS standards

Q: Describe the IEEE recommended structure for an SRS. How does IEEE 29148 extend IEEE 830?

Model answer points:

  • State IEEE 830 sections: introduction, overall description, specific requirements (interfaces, functional, performance, constraints, attributes), appendices.
  • Explain IEEE 29148 covers full RE process; stakeholder/system/software requirements levels.
  • List requirement attributes in 29148: ID, statement, rationale, source, priority, verification method.
  • Mention IEEE 729 glossary definition of requirement.
  • State benefits: complete template, contractual clarity, compliance, international alignment.
