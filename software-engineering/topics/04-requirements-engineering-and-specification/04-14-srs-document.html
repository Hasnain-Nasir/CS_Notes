4.14 Requirements Specifications / SRS Document

← Topic 4

A Software Requirements Specification (SRS) is the primary deliverable of requirements specification. It is a complete, formal description of the behaviour, functions, performance, and constraints of a software system. The SRS serves as the contract between customer and developer and as the baseline for design, testing, and acceptance.

Main sections of an SRS document

  • Introduction: Purpose of the document, scope of the product, definitions and acronyms, references, and overview of the remainder of the document.
  • General description: Product perspective (how the system fits into the larger environment), product functions at a high level, user characteristics, general constraints, and assumptions/dependencies.
  • Functional requirements: Detailed description of each function the system must perform, usually organised by feature or use case. Each requirement should be uniquely identified.
  • Interface requirements: User interfaces, hardware interfaces, software interfaces (APIs), and communication interfaces the system must support.
  • Performance requirements: Speed, throughput, response time, capacity, and availability targets.
  • Design constraints: Restrictions on design choices — standards, hardware limitations, technology mandates.
  • Non-functional requirements: Security, reliability, maintainability, portability, usability, and other quality attributes not covered above.
  • Schedule and budget constraints: Delivery milestones, resource limits, and cost boundaries imposed on the project (links to Topic 3 planning).
  • Appendices: Supporting material such as sample input/output, glossary, analysis models (DFD, ERD), and traceability matrices.

SRS audience

  • Customers and users review the SRS to confirm the system will meet their needs.
  • Project managers use it for planning, estimation, and change control.
  • Designers and developers implement the specified behaviour.
  • Testers derive test cases from each requirement for verification and acceptance testing.
  • Maintenance engineers refer to the SRS when modifying the system after delivery.

SRS and the SDLC

The approved SRS marks the end of the requirements phase in the SDLC and the input to the design phase. Changes after approval require formal change control (see §4.7).

Exam question — SRS structure

Q: What is an SRS document? List its main sections and explain the purpose of each.

Model answer points:

  • Define SRS as formal complete description of system behaviour, functions, performance, constraints.
  • List sections: introduction, general description, functional, interface, performance, design constraints, non-functional, schedule/budget, appendices.
  • Explain purpose of each section briefly.
  • Name audience: customers, PM, designers, testers, maintainers.
  • State SRS as requirements phase output and design phase input; link to change control.
