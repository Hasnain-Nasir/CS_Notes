11.3 Reliability Growth Models

← Topic 11

Software Reliability Growth Models (SRGMs) are mathematical models that describe how software reliability improves as faults are detected and removed during testing. They are introduced here as an overview; individual syllabus models are covered in §11.4 through §11.8.

Definition

  • An SRGM is a statistical model that relates the number of failures (or the failure rate) to the amount of testing effort — measured in execution time or number of test cases.
  • As testing progresses and faults are removed, the failure intensity decreases and reliability increases — this growth pattern is what the model captures.
  • SRGMs are black-box models: they use observed failure data without requiring knowledge of the internal program structure.
  • No single SRGM is universally valid — each model makes specific assumptions about fault detection and debugging behaviour.

What SRGMs represent

  • SRGMs plot reliability or failure rate (intensity) against time (execution hours) or against number of test cases executed.
  • The x-axis is cumulative testing effort; the y-axis is either the probability of failure-free operation (reliability) or the instantaneous failure rate λ(t).
  • A typical growth curve shows high failure intensity early in testing, declining as more faults are found and fixed.
  • The curve can be used to estimate when the software will reach a target reliability level (e.g. 0.99 over 100 hours).

Purposes of SRGMs

  • Predict reliability: Estimate the reliability the software will achieve after additional testing — e.g. "after 500 more test cases, reliability will reach 0.995."
  • Guide testing: Determine how much more testing is needed to reach a reliability target before release.
  • Allocate resources: Decide whether to continue testing, add staff, or release based on projected reliability versus the required threshold.
  • Identify problem areas: A model that fits poorly may indicate inconsistent debugging, incomplete fault removal, or heterogeneous fault populations.

Black-box models

  • Black-box SRGMs treat the software as an opaque system — only failure times or failure counts are observed, not code paths or module structure.
  • They assume faults are removed upon detection (perfect or imperfect debugging depending on the model).
  • Parameters are estimated from test data using Maximum Likelihood Estimation (MLE) or least-squares fitting.
  • White-box (structure-based) models use program complexity metrics but are less common in the syllabus SRGM set.

Failure data types

Data typeNameWhat is recordedExample
Type-I Time-between-failure (TBF) Exact time intervals between successive failures during execution Failures at 10 h, 25 h, 40 h → TBF = [10, 15, 15]
Type-II Failure-count Number of failures observed within fixed time intervals or after fixed test cases 3 failures in first 100 test cases, 2 in next 100 → counts = [3, 2]
  • Type-I data is more informative because exact failure times are recorded, but it requires continuous monitoring during execution.
  • Type-II data is easier to collect in batch testing but provides less precise timing information.
  • Different SRGMs use different data types — JM and SW use Type-I; GO and PNZ support both Type-I and Type-II.

Non-homogeneous Poisson process (NHPP)

  • Many SRGMs are based on the Non-Homogeneous Poisson Process (NHPP), where the failure intensity λ(t) varies with time rather than remaining constant.
  • In an NHPP, the expected number of failures by time t is given by the mean value function m(t) = ∫₀ᵗ λ(s) ds.
  • As faults are removed, λ(t) decreases and m(t) grows more slowly — modelling reliability growth.
  • The §3.11 Quasi Renewal Process is a related counting process where inter-failure intervals grow by a constant factor α > 1, modelling reliability growth during burn-in.

Duane and Coutinho models (brief)

  • The Duane model (1964) was originally developed for hardware reliability growth and plots cumulative failure rate against cumulative operating time on log-log axes — a straight line indicates reliability growth.
  • The Coutinho model extends Duane's approach to software, relating failure rate to cumulative test time with a power-law relationship.
  • Both are empirical curve-fitting models rather than stochastic process models — they are less detailed than JM, SW, GO, or PNZ but useful for quick trend analysis.

SRGM workflow — pseudocode

srgm_workflow.pseudo 
// Step 1: Collect failure data during testing
IF continuous monitoring THEN
  data = Type_I_time_between_failures()   // exact failure times
ELSE
  data = Type_II_failure_counts()       // failures per test interval
END IF

// Step 2: Select an appropriate SRGM
// Consider: data type, debugging assumptions, growth curve shape
model = SELECT_MODEL(data)   // e.g. JM, SW, GO, PNZ

// Step 3: Estimate model parameters via MLE or least squares
params = ESTIMATE_PARAMETERS(model, data)
// e.g. JM: N (initial faults), φ (hazard rate)
// e.g. GO: a (total faults), b (detection rate)

// Step 4: Validate model fit (goodness-of-fit test)
goodness = CHI_SQUARE_TEST(model, data, params)
IF goodness FAILS THEN
  TRY alternative model OR collect more data
END IF

// Step 5: Predict reliability at future testing effort
target_time = 500   // hours or test cases
R_predicted = model.RELIABILITY(target_time, params)
failure_rate = model.LAMBDA(target_time, params)

// Step 6: Decision — release or continue testing?
IF R_predicted >= reliability_target THEN
  RECOMMEND release
ELSE
  extra_effort = COMPUTE_EFFORT_FOR_TARGET(model, reliability_target)
  RECOMMEND continue testing for extra_effort units
END IF
Exam question — Reliability Growth Models

Q: Define SRGM. State its purposes. Differentiate Type-I and Type-II failure data. List the key syllabus models with their assumptions.

Model answer points:

  • SRGM = mathematical model relating failure rate/reliability to testing effort; black-box; reliability grows as faults are removed.
  • Purposes: predict reliability, guide testing, allocate resources, identify problem areas.
  • Type-I = time-between-failure (exact times); Type-II = failure-count per interval.
  • NHPP: failure intensity λ(t) varies with time; mean value function m(t) = ∫λ(s)ds.
  • JM (1972, binomial, perfect debugging); SW (1978, λ increases with time since debug); GO (1979, exponential NHPP); PNZ (imperfect debugging, S-shaped); Mills (error seeding).
  • Workflow: collect data → select model → estimate parameters → validate fit → predict → decide release.
