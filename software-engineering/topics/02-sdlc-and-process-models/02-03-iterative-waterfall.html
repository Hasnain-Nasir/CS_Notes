2.3 Iterative Waterfall Model

← Topic 2

The Iterative Waterfall Model is a practical refinement of the Classical Waterfall. It keeps the same phase order and the same emphasis on documentation, but introduces feedback loops: when an error or omission is found in a later phase, the team may return to an earlier phase, update the relevant documents, and then repeat the downstream work. This makes the process closer to what Winston Royce actually advocated for large systems — sequential structure plus controlled iteration when reality does not match the plan.

Iterative Waterfall is not the same as Incremental delivery (§2.8) or Agile sprints (§2.11). In Iterative Waterfall you still aim to build one complete product through the full phase chain; you only loop back to fix mistakes, not to release version 1, version 2, version 3 to customers.

Why the Iterative Waterfall model exists

Pure Classical Waterfall assumes perfection in every phase. In practice, requirements are misunderstood, designs omit edge cases, and code introduces defects. Without a formal way to go back, teams either (a) hide rework and patch code informally, or (b) fail acceptance tests. Iterative Waterfall legitimises controlled backtracking while keeping management visibility: each return trip updates baselines (SRS, DDS, test plans) and is recorded in change logs.

Phases (same as Classical Waterfall)

The phase names and deliverables are identical to §2.2: requirements → design → implementation → testing → deployment → maintenance. The difference is what happens when a phase discovers a problem that belongs to an earlier phase. Below are the standard feedback paths examiners expect you to describe.

Feedback paths and rules

Feedback is triggered by defects, reviews, or failed tests — not by casual preference to redesign. Each backward move should follow a change-control process (impact analysis, approval, document update).

1. Testing → Implementation (coding)

Most common loop. System or integration testing finds a coding bug or wrong module behaviour. Testers log defects; developers fix code, perform unit regression, and testing resumes. The SRS and design usually stay unchanged. This loop is narrow and cheap.

2. Testing → Design

Tests show that the design cannot meet performance, security, or correctness requirements — e.g., database schema causes deadlock under load. The team revisits the DDS, updates architecture or module interfaces, then re-implements affected modules and re-runs integration and system tests. Cost is moderate to high.

3. Testing or Design → Requirements

The most expensive loop. Stakeholders realise the SRS was wrong or incomplete: a missing business rule, illegal workflow, or misunderstood regulation. Analysts revise the SRS; design and code for affected parts are redone. Schedule and budget impact is large — this illustrates why requirements engineering is critical.

4. Implementation → Design (during coding)

While coding, developers discover the specified algorithm is inefficient or an interface is impractical. They request a design change before continuing. Iterative Waterfall allows returning to design without waiting until formal testing — good teams catch this early in code reviews.

5. Design → Requirements

During design reviews, architects find conflicting or ambiguous requirements. The SRS must be clarified before design can be completed. This loop prevents building the wrong structure on top of vague requirements.

Important exam point: feedback is typically local — you go back only to the phase where the fault originated, not always to the very first phase. Going from testing directly to requirements happens only for major specification errors.

Step-by-step scenario (how a loop works in practice)

Scenario — Online admission system:

  1. Requirements and design are signed off. Developers implement modules for application form and fee payment.
  2. During system testing, testers find that partial payments leave applications in an invalid state. A business rule was missing from the SRS.
  3. Project manager authorises feedback to requirements analysis. SRS is updated (v1.1) with refund/cancel rules.
  4. Design is updated for the payment module and database transactions.
  5. Affected code is rewritten. regression testing runs on payment and admission modules.
  6. When tests pass, flow continues to deployment. Total project time increases, but the product matches real policy.

Under Classical Waterfall, the same discovery at testing might be treated as “scope creep” or fixed with risky patches. Under Iterative Waterfall, the return path is explicit and documented.

Documentation and version control

Every feedback loop must update controlled documents:

  • SRS versions (v1.0, v1.1, …) with change history.
  • Design document revisions linked to requirement IDs.
  • Traceability matrix: requirement ↔ design ↔ test case ↔ code module.
  • Re-test evidence showing old defects are closed and no new ones introduced (regression).

Without version control, Iterative Waterfall degenerates into chaos — multiple developers working from different editions of the SRS. Configuration management (Topic 13) supports this model directly.

Advantages of the Iterative Waterfall Model

  • Error correction is built into the process — mistakes found in design or testing can be fixed at the source instead of patched silently in code.
  • More realistic than classical Waterfall — acknowledges that perfect upfront analysis is rare.
  • Preserves structure and accountability — phases, reviews, and sign-offs remain; suitable for organisations that cannot adopt Agile but need some flexibility.
  • Better quality at acceptance — major specification gaps can be closed before delivery, reducing catastrophic failure at go-live.
  • Aligns with Royce’s intent — iteration for risk reduction without abandoning documentation.
  • Supports regulated industries — auditors can see change requests and traceability when loops occur.

Disadvantages and limitations

  • Feedback is expensive — each return to requirements or design replays downstream work; frequent loops destroy schedule and budget. Not a licence for changing requirements every week.
  • Still no early product for the customer — unlike Incremental or Agile, users do not receive working subsets until the full cycle completes (unless combined with another model).
  • Document proliferation — multiple SRS/DDS versions; teams need strict CM discipline.
  • “Waterfall” stigma — managers may deny that loops are happening, leading to hidden technical debt.
  • Not for continuous requirement churn — if the market changes daily, Agile/Spiral/Prototype fit better.
  • Testing still relatively late — compared to V-Model or TDD, test planning is not paired at every micro-step unless the organisation adds extra practices.

When to use the Iterative Waterfall Model

  • Large projects with mostly stable requirements but some risk of discovery during design/testing.
  • Organisations with mature process culture: change control boards, traceability, formal reviews.
  • Contracts that require Waterfall-style reporting but allow approved change requests.
  • Systems where failures must be traced to document versions (finance, healthcare, aerospace documentation).

When NOT to use

  • Requirements evolve continuously (use Agile or evolutionary models).
  • Need frequent releases to market (Incremental, RAD).
  • High UI uncertainty. Users must see prototypes early (Prototype model, §2.4).
  • High technical risk needing risk-driven cycles (Spiral model, §2.5).

Combining with other ideas

Iterative Waterfall is often used inside each increment of an Incremental model, or as the internal process of a government phase-gate project. It does not replace risk analysis (Spiral) or early prototypes (Prototype model) — those can be added on top where needed.

Real-world example

Example — Hospital information system upgrade: Core workflows (patient registration, billing) are regulated and specified in a national health data standard. The hospital uses Iterative Waterfall: baseline SRS from the ministry, design review with clinicians, then development. During integration testing, clinicians find that emergency admissions bypass normal registration — the SRS lacked an exception flow. The team loops back to requirements, updates design, recodes two modules, and re-tests. The project is late by six weeks but passes accreditation. A classical “no feedback” approach would have shipped unsafe workarounds.

Exam question — Iterative Waterfall

Q: Explain the Iterative Waterfall model. How does it differ from the Classical Waterfall model? Illustrate with a diagram showing feedback paths. Discuss advantages, disadvantages, and suitable application areas with an example.

Model answer outline:

  • Define Iterative Waterfall as using the same phases as classical Waterfall but allowing controlled feedback when later phases discover errors in earlier work.
  • Explain that pure Waterfall is unrealistic and that Royce recommended iteration with updated documents.
  • Draw six phases and show backward paths from testing to coding, design, or requirements, and from design to requirements, explaining when each path is used.
  • Compare classical and iterative Waterfall using a table (one-way flow versus feedback, document versions, delivery timing, realism).
  • Clarify that Iterative Waterfall is not the same as Incremental delivery or Agile continuous change.
  • State that feedback is triggered by defects or reviews, follows change control, updates the SRS and DDS, and requires regression testing.
  • Walk through a scenario where testing finds a missing requirement, the SRS is updated, design and code are revised, and testing is repeated.
  • List advantages and disadvantages, suitable contexts (for example a hospital system), and when other models are preferable.
