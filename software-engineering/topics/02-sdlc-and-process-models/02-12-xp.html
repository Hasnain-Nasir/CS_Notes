2.12 Extreme Programming (XP)

← Topic 2

Extreme Programming (XP), introduced by Kent Beck, is an Agile methodology that takes proven practices to extreme levels. XP focuses on technical excellence, frequent releases, and close customer pairing with developers.

Core XP practices

  • Pair programming. Two developers share one workstation: the driver writes code while the navigator reviews and thinks ahead, and they swap roles frequently.
  • Test-Driven Development (TDD). The developer writes a failing test first, writes the minimal code to pass it, and then refactors (the Red, Green, Refactor cycle).
  • Continuous Integration (CI). The team merges code into a shared repository several times a day and runs automated build and test on each merge.
  • Collective code ownership. Any member of the team may improve any part of the codebase.
  • Simple design. The team builds only what is needed now, following the “You Aren’t Gonna Need It” (YAGNI) principle.
  • Refactoring. The team continuously improves code structure without changing external behaviour.
  • On-site customer. A user representative is available full-time to answer questions and clarify requirements.
  • Small releases. The team delivers small, deployable increments frequently rather than waiting for one large release.

Advantages and disadvantages

  • Advantages. XP tends to produce high code quality, fewer defects through TDD and pair review, rapid feedback, and strong communication within the team.
  • Disadvantages. Pair programming increases staffing cost on coding tasks, the approach needs highly skilled developers, an on-site customer is often impractical, and XP is less suited to very large teams or fixed-price contracts.
Exam question — Extreme Programming

Q: Describe Extreme Programming. Explain pair programming, TDD, and Continuous Integration with advantages and disadvantages.

Model answer points:

  • Define XP as an Agile method introduced by Kent Beck that takes good practices to an extreme level for quality and speed.
  • Explain pair programming as two developers sharing one keyboard for real-time review and knowledge sharing.
  • Explain TDD as writing a test first, then code, then refactoring, which improves coverage and keeps design simple.
  • Explain continuous integration as frequent merges with automated tests that catch defects early.
  • State advantages (quality and fewer defects) and disadvantages (cost of pairing, need for skilled staff and an available customer).

MCQ (GFG style): In TDD, the first step is — (A) Write a failing test.
