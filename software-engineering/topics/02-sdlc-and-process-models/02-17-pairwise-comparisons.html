2.17 Pairwise Model Comparisons

← Topic 2

Exams frequently ask "difference between X and Y" questions. Below are the four most common pairwise comparisons with tabular answers.

Waterfall vs Incremental

AspectWaterfallIncremental
DeliveryWaterfall delivers the complete product in a single release at the end.The Incremental model delivers multiple partial releases throughout the project.
FeedbackCustomer feedback arrives late, after the full system is built.Customer feedback arrives early, after each increment is delivered.
RequirementsWaterfall expects requirements to be complete before development starts.The Incremental model partitions requirements by priority across increments.
RiskWaterfall carries a high risk of building the wrong product if direction was wrong from the start.The Incremental model lowers that risk because early increments validate the project direction.
ArchitectureWaterfall designs the architecture once for the whole system.The Incremental model must plan for extensibility upfront so later increments can be added safely.

Waterfall vs Spiral

AspectWaterfallSpiral
IterationsWaterfall usually runs as a single pass, or allows feedback loops only in its iterative variant.The Spiral model runs multiple risk-driven cycles until the product is complete.
Risk analysisWaterfall treats risk analysis as implicit and often discovers problems late.The Spiral model makes risk analysis explicit in every cycle.
CostWaterfall is lower in cost for simple projects with stable requirements.The Spiral model is higher in cost because it uses multiple cycles and prototypes.
PrototypingPrototyping is not built into classical Waterfall.The Spiral model uses prototypes whenever they help resolve risks.
Best forWaterfall is best for small projects with stable requirements.The Spiral model is best for large, complex, high-risk projects.

Agile vs V-Model

AspectAgileV-Model
FlexibilityAgile is very flexible and welcomes changing requirements throughout the project.The V-Model has low flexibility because requirements and test plans are fixed early.
DocumentationAgile keeps documentation minimal and prioritises working software.The V-Model requires extensive test and design documentation for each phase.
TestingTesting runs continuously throughout every sprint.Testing is planned in pairs that align with each development phase.
Customer roleThe customer collaborates with the team daily or weekly.The customer is involved mainly during requirements definition and final acceptance.
Best forAgile is best for dynamic requirements and startup-style products.The V-Model is best for regulated and safety-critical systems.

V-Model vs Waterfall

AspectV-ModelWaterfall
TestingThe V-Model plans testing from the start and pairs each development phase with a matching test phase.Classical Waterfall concentrates testing mainly after the coding phase is complete.
TraceabilityThe V-Model provides strong traceability from each requirement to its corresponding tests.Waterfall provides weaker traceability unless the team maintains it manually.
StructureThe V-Model uses the same sequential development phases as Waterfall but adds a parallel testing arm.Waterfall follows linear phases without formally paired test stages.
Verification/ValidationThe V-Model makes an explicit distinction between verification and validation.Waterfall does not formally distinguish verification from validation.
Use caseThe V-Model is used for medical devices, aerospace, and banking systems that need strict compliance.Waterfall is used for general projects with fixed requirements and conventional documentation needs.
Exam question — Pairwise comparisons

Q: Differentiate: (a) Waterfall vs Incremental, (b) Waterfall vs Spiral, (c) Agile vs V-Model, (d) V-Model vs Waterfall.

Model answer points:

  • Use the four comparison tables above and write at least four or five complete comparison points for each pair.
  • Explain key contrasts such as delivery timing, risk, flexibility, testing approach, and the customer’s role in the project.
  • Always state when each model is preferred, not only how the two models differ.

MCQ (GFG style): Which model is most suitable for high-risk, large projects? — (C) Spiral Model.
