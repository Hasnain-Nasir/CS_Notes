6.5 Lines of Code (LOC)

← Topic 6

Lines of Code (LOC) is the oldest and simplest size-oriented software metric. A line of code is any line of program text that is not a comment or blank line — including header lines, variable declarations, and both executable and non-executable statements.

LOC is a direct (white-box) measurement: it counts the actual volume of source code. In project management, size is often expressed as KLOC (thousands of lines of code). LOC feeds effort estimation through productivity formulas covered in §3.5.

What counts as a line of code

  • LOC includes lines with variable declarations, executable statements, and non-executable statements such as type definitions.
  • LOC excludes blank lines and comment lines.
  • Some counting methods also exclude header/include lines — always follow the counting convention stated in the exam question.
  • A single physical line may contain multiple statements; it still counts as one LOC unless the question specifies statement counting.

Features of LOC as a metric

  • Ease of computation: LOC is simple to calculate automatically or manually with minimal effort.
  • Easy to understand: Stakeholders — even non-technical managers — grasp "lines of code" intuitively.
  • Change tracking: Variations in LOC over time show whether a codebase is growing or shrinking during development and maintenance.
  • Benchmarking: LOC can compare different versions of the same program to see how changes affect total code size.
  • Productivity measure: LOC per person-month is widely used to compare developer or team productivity within the same language and coding standard.
  • Limitation — complexity: Two programs with the same LOC can have very different logical complexity; LOC measures volume, not quality.
  • Limitation — language dependency: LOC is meaningful only when comparing projects in the same language with similar coding standards.

LOC counting example

Consider a simple selection sort function:

selSort.c 
void selSort(int x[], int n) {
  // Sorts an array in ascending order
  int i, j, min, temp;
  for (i = 0; i < n - 1; i++) {
    min = i;
    for (j = i + 1; j < n; j++)
      if (x[j] < x[min])
        min = j;
    temp = x[i];
    x[i] = x[min];
    x[min] = temp;
  }
}
  • Total physical lines including comments = 13 LOC.
  • After excluding comments and blank lines = 12 LOC.

Exam questions often specify whether comments and blanks are excluded — read the question carefully before counting.

LOC in project estimation

  • LOC is used in direct (white-box) decomposition estimation: break the system into modules, count LOC per module, sum to get total size.
  • Key relationships (from §3.5):
FormulaExpression
EffortEffort = Size ÷ Productivity
ProductivityProductivity = Size ÷ Effort (e.g., LOC per person-month)
CostCost = Effort × Pay rate
DurationDuration = Effort ÷ Team size
Team sizeTeam size = Effort ÷ Duration

PDF worked examples

Example 1 — Banking application modules

  • User Interface = 5,400 LOC; Database = 8,500 LOC; Server = 7,300 LOC; Client = 5,500 LOC.
  • Total size = 5,400 + 8,500 + 7,300 + 5,500 = 26,700 LOC.
  • Productivity = 870 LOC/month; salary = $1,500/month per developer.
  • Effort = 26,700 ÷ 870 ≈ 30.7 person-months; cost = 30.7 × 1,500 ≈ $46,000.

Example 2 — Three modules in KLOC

  • M1 = 14.4 KLOC; M2 = 21.5 KLOC; M3 = 8.4 KLOC; total = 44.3 KLOC.
  • Cost of writing 1 KLOC = $50 → total cost = 44.3 × 50 = $2,215.
  • Productivity = 4 KLOC/month → effort = 44.3 ÷ 4 ≈ 11.1 person-months.

Example 3 — PERT-style expected size

  • M1 = 4,600 LOC (optimistic); M2 = 6,900 (most likely); M3 = 8,600 (pessimistic).
  • Expected size = (Soptimistic + 4 × Slikely + Spessimistic) ÷ 6.
  • Per module: (4600 + 4×6900 + 8600) ÷ 6 = (4600 + 27600 + 8600) ÷ 6 ≈ 6,867 LOC for M2-style calculation; sum all three modules for total expected size.

LOC and empirical models

  • Most parametric estimation models — especially COCOMO — accept size in KLOC as primary input.
  • Basic COCOMO: DE = a × (KLOC)b; DD = c × (KLOC)d; Team size = DE ÷ DD.
  • Other KLOC-based models: Walston-Felix E = 5.2 × (KLOC)0.91; Bailey-Basili; Simple Boehm; Doty model.
  • Research shows a rough correlation between LOC and total development cost, duration, and number of defects — lower LOC often means lower cost and fewer bugs, but this depends on quality of code, not volume alone.

LOC vs Function Points

AspectLOCFunction Point (FP)
Metric typeSize-oriented (white-box / direct)Function-oriented (black-box / indirect)

In practice, FP is preferred for business systems because functional size remains constant regardless of implementation language. LOC remains useful for comparing productivity within a single language and team. See §6.4 for full FP calculation.

Exam questions — Lines of Code

Q1 (UGC-NET): LOC and FP measure — (B) Size of Software.

Q2: Differentiate LOC and Function Point with four points.

Q3: A project has modules of 5400, 8500, 7300, and 5500 LOC. Productivity is 870 LOC/month and pay is $1500/month. Find effort and cost.

Model answer points:

  • LOC = non-comment, non-blank program lines; KLOC = thousands of LOC; white-box/direct metric.
  • Effort = Size/Productivity; Cost = Effort × Pay; used in COCOMO and decomposition estimation.
  • LOC is language-dependent; FP is language-independent and user-oriented.
  • Example: total 26,700 LOC; effort ≈ 30.7 PM; cost ≈ $46,000 at given productivity and pay.
