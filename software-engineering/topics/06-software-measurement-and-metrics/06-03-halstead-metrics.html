6.3 Halstead's Software Science / Metrics

← Topic 6

Halstead's Software Science (also called Halstead's Software Metrics) was proposed by Maurice Halstead in 1977. It treats a computer program as a collection of tokens classified as either operators (symbols that perform actions) or operands (entities on which actions are performed). All Halstead metrics are derived from four basic counts.

Basic token counts

SymbolMeaning
n1Number of distinct operators
n2Number of distinct operands
N1Total occurrences of operators
N2Total occurrences of operands

Core Halstead metrics and formulas

MetricFormulaMeaning
Program length (N)N = N1 + N2Total operator and operand occurrences
Vocabulary (n)n = n1 + n2Total distinct operators and operands
Volume (V)V = N × log2(n)Program size in bits; reflects information content
Difficulty (D)D = (n1 / 2) × (N2 / n2)How hard the program is to understand or write
Level (L)L = 1 / D = n2 / (n1 × N2) × 2Abstraction level; higher L = simpler program
Effort (E)E = V × D = V / LMental effort to implement the program
Time (T)T = E / (S × f)Estimated implementation time in seconds
Estimated length (N̂)N̂ = n1·log2(n1) + n2·log2(n2)Predicted program length from vocabulary
Potential minimum volume (V*)V* = (2 + n2*) × log2(2 + n2*)Smallest possible volume for the problem
Intelligence content (I)I = V / DAmount of intelligence expressed in the program
Estimated errorsErrors ≈ V / 3000Predicted number of defects in the program

For time calculation, Halstead uses the Stroud number S = 18 (moments per second of human decision-making) and f = 60 (seconds-to-minutes conversion factor).

Counting rules for C programs

  • Comments, identifier declarations, and function declarations are not counted.
  • All variables and constants are counted as operands.
  • Function calls are counted as operators.
  • Control statements (if, else, for, while, do, switch, case, return, break, continue) are counted as operators.
  • Brackets, commas, semicolons, and terminators are counted as operators.
  • GOTO is an operator; its label is an operand.
  • Unary and binary occurrences of +, -, and * are counted separately.
  • In array[index], the array name and index are operands; [ ] is an operator.
  • In struct.member or struct->member, struct name and member name are operands; . and -> are operators.
  • Preprocessor directives (#include, #define, etc.) are ignored.
  • Global variables used in multiple modules count as multiple occurrences of the same operand.
  • Local variables with the same name in different functions count as unique operands.

Worked example — sorting function

Consider this C sorting function used in Halstead analysis:

sort.c 
int sort(int x[], int n) {
  int i, j, save, im1;
  /* Sorts array x in ascending order */
  if (n < 2) return 1;
  for (i = 2; i <= n; i++) {
    im1 = i - 1;
    for (j = 1; j <= im1; j++)
      if (x[i] < x[j]) {
        save = x[i];
        x[i] = x[j];
        x[j] = save;
      }
  }
  return 0;
}

After applying counting rules:

Distinct (n)Total (N)
Operatorsn1 = 14N1 = 53
Operandsn2 = 10N2 = 38
  • Program length: N = N1 + N2 = 53 + 38 = 91.
  • Vocabulary: n = n1 + n2 = 14 + 10 = 24.
  • Volume: V = N × log2(n) = 91 × log2(24) ≈ 417.23 bits.
  • Estimated length: N̂ = n1·log2(n1) + n2·log2(n2) ≈ 86.51.
  • Difficulty: D = (n1/2) × (N2/n2) = (14/2) × (38/10) = 26.6 (≈ 37.03 with n2* adjustment in full analysis).
  • Effort: E = V × D ≈ 15,400 (mental effort units).
  • Time: T = E / (18 × 60) ≈ 610 seconds (~10 minutes).
  • Predicted errors: V / 3000 ≈ 0.14 defects.

Interpretation of Halstead metrics

  • As program volume increases, difficulty rises and the program level decreases — redundant operands and failure to use higher-level constructs increase both volume and difficulty.
  • Higher language level (L) means less effort is needed — Pascal requires less effort than Assembler for the same algorithm.
  • Volume, program length (N), and lines of code are shown to be linearly related — all are valid for comparing relative program size.
  • Halstead metrics can be computed automatically from source code without full structural analysis of the program architecture.
  • They predict maintenance effort, error rate, and scheduling needs for a module.

Limitations to remember for exams

  • Halstead metrics require the complete source code — they cannot estimate size before coding is done.
  • They assume all operators and operands are equally important, which is not always true in practice.
  • They focus on code complexity and effort — they do not directly measure reliability, usability, or maintainability.
  • They are less useful as a predictive estimating model for entire projects compared to function points or COCOMO.
Exam question — Halstead metrics

Q: Define Halstead's Software Science. Write the formulas for program length, vocabulary, volume, difficulty, effort, and time. What do n1, n2, N1, and N2 represent?

Model answer points:

  • Halstead treats a program as tokens — operators (actions) and operands (data acted upon).
  • n1 = distinct operators; n2 = distinct operands; N1 = total operator occurrences; N2 = total operand occurrences.
  • N = N1+N2; n = n1+n2; V = N·log2(n); D = (n1/2)·(N2/n2); E = V·D; T = E/(18×60).
  • Volume measures size in bits; difficulty measures how hard code is to handle; effort = mental work; time uses Stroud number S=18.
  • Predicted errors ≈ V/3000. Requires complete code; size-oriented (white-box) metric.
