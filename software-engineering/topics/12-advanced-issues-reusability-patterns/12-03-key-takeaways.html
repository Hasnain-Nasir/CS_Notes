Key takeaways — exam revision

← Topic 12
  • 12.1 Reusability: Software reuse = building new systems from existing artifacts and knowledge rather than from scratch; addresses software crisis; umbrella activity parallel to SDLC.
  • Benefits: Reduced development time/cost; better tested and more reliable components; consistency across products; faster response to new requirements.
  • Types of reuse: Code, design, specification, component, framework, pattern reuse — each operates at a different lifecycle level.
  • Reusable component traits: Generic, modular, low coupling/high cohesion, well-documented, thoroughly tested, version-controlled, configurable.
  • DRY principle: Every piece of knowledge has a single authoritative representation; avoids duplicated logic and inconsistent maintenance.
  • Component-Based Architecture (CBA): Software assembled from reusable, independently deployable components with well-defined interfaces; promotes modularity and flexibility.
  • Reuse-Oriented Model (ROM) — 4 steps: (1) Identify reusable components, (2) Understand all components, (3) Modify components, (4) Integrate into new system. Requires component catalogue; risk if library is insufficient.
  • Reuse Maturity Model: L1 single-project source pool; L2 multi-project source copy (maintenance nightmare); L3 ad hoc binary reuse (utility in own project); L4 controlled binary reuse with version tracking (Reuse/Release Equivalence Principle).
  • Reuse vs re-engineering vs reverse engineering: Reuse = use existing artifacts for new systems; Re-engineering = improve legacy system without changing core purpose; Reverse engineering = recover design/requirements from existing code (8 steps).
  • 12.2 Design Patterns: Reusable proven solutions to recurring design problems; GoF 1994; language-independent; design-level reuse (not ready-made code).
  • GoF pattern template: Name, problem, solution, consequences.
  • Three categories: Creational (object creation — Singleton, Factory, Abstract Factory, Builder, Prototype); Structural (composition — Adapter, Bridge, Composite, Decorator, Facade, Flyweight, Proxy); Behavioral (communication — Observer, Strategy, Command, State, Template Method, etc.).
  • Singleton: One instance only; global access point; private constructor; used for DB manager, config, logging.
  • Factory Method: Interface for creating objects; subclasses decide concrete class; decouples creation from use; supports Open/Closed Principle.
  • Observer: One-to-many dependency; subject notifies all observers on state change; publish-subscribe; loosely coupled.
  • Strategy: Family of interchangeable algorithms encapsulated in strategy classes; selected/switched at runtime; eliminates complex if-else chains.
  • Patterns vs code reuse: Patterns = design-level blueprint; code reuse = implementation-level modules; complement each other; overuse without need adds complexity.
  • Links: Patterns extend OOD (coupling/cohesion, §5.10); documented with UML class diagrams (§5.16); patterns are design-level form of reusability (§12.1).
