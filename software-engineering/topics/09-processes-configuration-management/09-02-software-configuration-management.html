9.2 Software Configuration Management

← Topic 9

Software Configuration Management (SCM) is the process of defining and implementing a standard configuration for a software project so that the product remains in a known, controlled state throughout development. SCM tracks and manages the emerging product and its versions, ensures all stakeholders know what is being designed, developed, built, tested, and delivered, and enables requirements to be traced through to the final software product.

Definition

  • SCM can be defined as the process of identifying, organizing, and controlling changes to software work products throughout the lifecycle.
  • It manages not only source code but also documentation, data, tools, and hardware used in development.
  • SCM includes revision control (tracking versions of each item) and establishment of baselines (formally approved snapshots of the product).
  • Primary benefits include easier setup and maintenance, less downtime, better enterprise integration, more reliable backups, and maximised productivity by minimising mistakes.
  • SCM is an umbrella activity — it runs in parallel with the entire SDLC, not just during coding — see §9.1.

Configuration Items (CIs)

  • A Configuration Item (CI) is any work product that is placed under SCM control — source code files, design documents, test plans, executables, libraries, and user manuals.
  • Each CI is uniquely identified, versioned, and tracked so that changes can be traced to their origin.
  • Configuration identification determines the scope of the software system — what items exist and which of them will be managed.
  • Relationships among CIs are established so that the impact of a change to one item on others can be assessed.

Baselines

  • A baseline is a formally accepted version of a software configuration item — designated and fixed at a specific point in time.
  • Per IEEE 610.12-1990, a baseline is a specification or product that has been formally reviewed and agreed upon and serves as the basis for further development; changes occur only through formal change control.
  • A baseline is a set of mutually consistent CIs that have been formally reviewed and agreed upon at a given milestone.
  • Once a baseline is set, it can only be changed through formal change control procedures — not by ad hoc edits.
  • Baselines freeze a known-good state so that other teams can integrate against a stable reference while development continues elsewhere.
  • Common baseline types include: functional, allocated, design, product, operational, acceptance test, integration test, and unit test baselines.
  • The baseline process: document work products → review and fix issues → approve baseline → evaluate and approve all subsequent changes formally.

What SCM controls

  • Source code — all program files under version control.
  • Documentation — SRS, design documents, test plans, user manuals, release notes.
  • Executables and libraries — compiled binaries, DLLs, and third-party components.
  • Tools and environments — compilers, build scripts, configuration files, and deployment settings.
  • SCM ensures the system is always in a known and stable state — every change is recorded, and any version can be reproduced.

SCM and traceability

  • Through SCM, design requirements can be traced to the final software product — each requirement links to design elements, code modules, test cases, and delivered features.
  • SCM supports reproducibility — a software system can be replicated at any stage of development for testing, debugging, or deployment.
  • SCM enables parallel development — multiple developers work on different branches simultaneously without corrupting the main codebase.
  • Effective SCM links code modifications to reported issues, making bug tracking more accurate and auditable.
Exam question — Software Configuration Management

Q: Define Software Configuration Management. Explain Configuration Items and baselines with their role in the SCM process.

Model answer points:

  • SCM = process of defining/implementing standard configuration; tracks product and versions; ensures stakeholder visibility and traceability.
  • Includes revision control + baselines; manages code, docs, tools, executables.
  • CI = any work product under SCM control, uniquely identified and versioned.
  • Baseline = formally accepted CI snapshot; mutually consistent CIs; fixed at a point in time; changed only via formal change control.
  • Baseline types: functional, design, product, test baselines, etc.
  • Benefits: known stable state, reproducibility, parallel development, traceability, bug tracking.
