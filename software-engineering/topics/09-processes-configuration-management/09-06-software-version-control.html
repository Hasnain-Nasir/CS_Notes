9.6 Software Version Control

← Topic 9

Software version control is a set of procedures and tools for managing versions of configuration items — source code, documentation, data, and executables. It allows teams to track every change, collaborate without overwriting each other's work, and revert to earlier versions when something goes wrong.

What is version control?

  • Version control keeps track of changes in computer programs and related work products over time.
  • It lets team members see what changes were made, when, and by whom, and go back to an older version if a change introduces a defect.
  • A version is a collection of software configuration items (SCIs) that together represent a defined state of the product — it may contain variants for different platforms or customers.
  • Version control is one of the five SCM activities — see §9.4 SCM Activities.

Importance of version control

  • Collaboration: Multiple developers work on the same project simultaneously without corrupting each other's changes.
  • Code management and history: Every change is recorded, providing a complete history that supports debugging and auditing.
  • Rollback capability: If a new change breaks the system, the team can revert to any previous known-good version instantly.
  • Quality assurance: Controlled releases ensure that only tested, approved code reaches production environments.
  • Reusability: Prior versions and components can be retrieved and reused in new projects or branches.
  • Branching and merging: Parallel development paths (features, hotfixes, releases) are managed and merged systematically.
  • Project management: Version milestones align with project schedules and deliverable tracking.
  • CI/CD integration: Version control integrates with continuous integration and deployment pipelines for automated build and test cycles.

Four version control sub-activities

  1. Identifying new versions: Each time changes are made and saved, the version control system creates a new unique version. Prior versions (v0, v1, v2…) are stored for reference and rollback.
  2. Numbering scheme: Versions follow a structured naming convention — typically X.Y.Z where X = whole SCI or major release, Y = component-level change, Z = minor correction within a component (e.g. 1.0 → 1.1 → 1.1.1 → 1.2 → 2.0).
  3. Visibility: The version number appears in the document frame or below the title so all team members know which version they are viewing or editing.
  4. Tracking: A version development graph records the history of changes — who made each change, when, and what was modified — maintaining accountability across the team.

Version evolution example

  • After initial release at version 1.0, minor corrections produce 1.1, then 1.1.1 and 1.1.2.
  • A major update creates version 1.2; development of the 1.x line continues through 1.3 and 1.4.
  • A significant architectural change creates a new evolutionary path — version 2.0 — while version 1.x may still be supported for existing customers.
  • Both evolutionary paths can be maintained simultaneously, each with its own branch in the version control system.

Version control tools — Git workflow

Modern projects use distributed version control systems such as Git. A typical workflow:

git_workflow.sh 
# Initialise a new repository
git init

# Stage and commit changes (creates a new version snapshot)
git add .
git commit -m "Initial baseline v1.0"

# Create a feature branch for parallel development
git checkout -b feature/login-module

# Work on feature, commit incrementally
git add login.py
git commit -m "Add login module v1.1"

# Merge feature branch back to main after review
git checkout main
git merge feature/login-module

# Tag a release version
git tag v1.1.0

# View version history
git log --oneline
  • Centralised VCS (e.g. SVN): a single server holds the repository; developers check out and check in files.
  • Distributed VCS (e.g. Git): every developer has a full copy of the repository history, enabling offline work and flexible branching.
  • Version control repositories serve as the project database referenced in the change control process — CIs are checked out, modified, and checked in through the VCS.
Exam question — Software Version Control

Q: What is software version control? List its sub-activities and explain the X.Y.Z numbering scheme.

Model answer points:

  • Version control = procedures + tools to manage versions of SCIs (code, docs, data).
  • Benefits: collaboration, history/rollback, QA, reusability, branching/merging, CI/CD integration.
  • Four sub-activities: identify new versions, numbering scheme (X.Y.Z), visibility, tracking (version graph).
  • X = major release/whole SCI; Y = component change; Z = minor correction within component.
  • Multiple evolutionary paths (1.x and 2.x) supported simultaneously.
  • Tools: Git (distributed), SVN (centralised); check-out/modify/check-in in change control.
