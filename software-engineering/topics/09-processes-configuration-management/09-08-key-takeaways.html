Key takeaways — exam revision

← Topic 9
  • 9.1 Overview: SCM = umbrella activity parallel to entire SDLC; part of SPM. Needed for multi-person projects, concurrent versions/branches, changing requirements, multi-platform deployment. Without SCM: inconsistency, concurrent-access conflicts, unstable environment, no status accounting, variant chaos.
  • 9.2 SCM definition: Process of defining/implementing standard configuration; tracks product and versions; ensures stakeholder visibility and traceability. Includes revision control + baselines.
  • Configuration Item (CI): Any work product under SCM control — code, docs, executables, tools — uniquely identified and versioned.
  • Baseline: Formally accepted CI snapshot; mutually consistent CIs; fixed at a point in time; changed only via formal change control. Types: functional, design, product, test baselines.
  • 9.3 Objectives: Classic 3: identify configuration at points in time; systematically control changes; maintain integrity & traceability. Also: collaboration, version control, replication, stakeholder visibility, remote admin, reduced downtime, reliable backups.
  • 9.4 SCM Activities (5): (1) Configuration identification/establishment, (2) Version control, (3) Change control, (4) Status accounting, (5) Configuration audit & reporting.
  • 9.5 Change Control: CR → evaluate (merit, side effects, impact, cost) → change report → CCB decision → ECR (if approved) → check-out → modify → test → check-in → new version. Roles: requestor, PM, CCB, reviewer. No silent changes; revise plan when significant scope/schedule changes.
  • 9.6 Version Control: Procedures + tools for managing SCI versions. Sub-activities: identify versions, numbering (X.Y.Z), visibility, tracking graph. X = major release; Y = component; Z = minor fix. Tools: Git (distributed), SVN (centralised). Benefits: collaboration, rollback, branching/merging, CI/CD.
  • 9.7 System SCM: Integrated SCM at whole-system level; manages code, docs, executables, environments. Essential part of all PM activities. Benefits: bug tracking, CI/CD, risk management, reproducibility, parallel development, replication, team coordination. Links to SQA, CMM Level 2.
  • CCB: Change Control Board — approves/rejects/defer changes; logs requests; notifies requestor.
  • ECR: Engineering Change Request — generated for each approved change; describes change, constraints, audit criteria.
  • Status accounting: Tracks what each version contains; records who/when/why for every change.
  • Configuration audit: Verifies product satisfies baseline; confirms completeness, correctness, consistency; complements FTR.
