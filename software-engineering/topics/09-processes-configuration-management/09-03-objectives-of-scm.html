9.3 Objectives of SCM

← Topic 9

The objectives of Software Configuration Management define what SCM is designed to achieve for a project and organisation. SCM is an umbrella activity applied throughout the software process — it manages and tracks the emerging product and its versions, identifies and controls the configuration of software, hardware, and tools, and ensures all stakeholders know what is being designed, developed, built, tested, and delivered. Objectives span classic software-engineering goals (integrity, traceability, controlled change), system-level goals (collaboration, replication), and operational standards goals (reliable environments, reduced downtime, multi-user support).

SCM objectives connect directly to the definition of SCM in §9.2 and are implemented through the activities described in §9.4 SCM Activities.

Three classic configuration management objectives

Configuration management (as part of overall change management) has three major purposes that form the foundation of all SCM work:

  • Identify product configuration at points in time: SCM records the exact state of all configuration items at each milestone so the team knows precisely what was built, tested, and delivered at any given moment.
  • Systematically control changes to the configuration: Every modification follows a formal process — request, evaluation, approval, implementation, verification — rather than ad hoc edits that bypass team awareness.
  • Maintain integrity and traceability throughout the product lifecycle: SCM ensures the product remains internally consistent and that every change can be traced from requirement through design, code, test, and delivery.
  • These three objectives allow large teams to work together in a stable environment while still providing the flexibility required for creative development work.
  • When developing and maintaining a project, changes to parameters are inevitable — these objectives ensure changes are managed rather than allowed to destabilise the product.

Key system-level SCM objectives

  • Control the evolution of software systems: Changes are properly planned, tested, and integrated into the final product rather than applied unpredictably or without impact analysis.
  • Enable collaboration and coordination: All team members work from the same version of each work product, with conflicts detected and resolved systematically through version control tools.
  • Provide version control: Teams manage and track different versions of the system, support multiple release paths, and revert to earlier versions when a change introduces defects.
  • Facilitate replication and distribution: Software systems can be replicated to test, production, and customer environments with confidence that the configuration is identical at every site.
  • Ensure stakeholder visibility: All people involved in the software process know what is being designed, developed, built, tested, and delivered at every stage — no hidden or undocumented work products exist.
  • Reduce project vulnerability: By fixing key deliverables at baselines and controlling all subsequent changes, SCM reduces the weakness that leads to uncontrollable modifications — see §9.2 Baselines.

Objectives of SCM standards (operational)

Beyond software-project goals, SCM standards also aim to achieve reliable, interchangeable, and manageable computing environments:

Remote system administration

  • Configuration standards include the necessary software and privileges for remote administration tools on each machine.
  • A correctly installed and configured remote administration client on each workstation enables central management of the entire development network.
  • Remote tools can check virus protection versions, verify machine configuration, and provide remote help-desk functionality without physical access to each machine.

Reduced user downtime

  • A standard configuration makes systems completely interchangeable — if one machine fails, an identical replacement can be deployed with minimal disruption.
  • On an unrecoverable error, an identical new system can be dropped into place; user data is transferred from the old machine or restored from the most recent backup.
  • The goal is that the user experiences little change in the system interface and installed software after a replacement or recovery.

Reliable data backups

  • Using a standard directory for user data allows the backup system to selectively backup a small portion of each machine, reducing network traffic and tape usage.
  • A divided directory structure between system data and user data is one of the main goals of configuration standards — backups become predictable and efficient.
  • Version-controlled repositories also serve as authoritative backups of all project artefacts between scheduled backup cycles.

Easy workstation setup

  • A standardised configuration streamlines the process of setting up new development machines and ensures that vital components are always present.
  • When multiple machines are configured to the same standard, most setup steps can be automated, reducing onboarding time for new team members.
  • Consistent environments prevent "works on my machine" problems caused by mismatched tool versions or missing dependencies.

Multi-user support

  • System configuration is designed so multiple users can share the same workstation without interfering with each other's settings or data.
  • Some software packages do not support completely independent settings for all users, but users can still have independent data directories under a structured layout.
  • A well-designed directory structure does not impose limits on the number of independent users a system can support.

Remote software installation

  • Factory-default software installations are suitable for single-user machines but lead to non-uniform configurations across a collection of development machines.
  • A good configuration standard installs software into specific directory areas to logically divide software on the disk — e.g. separating development tools, runtime libraries, and user applications.
  • Universal installation scripts make it easy to identify installed components and automate installation procedures across many machines simultaneously.
  • When software is installed into specific directories, maintenance and upgrading of running software becomes less complex and more predictable.

Change control objective

  • Change control is part of the overall change management approach — its objective is to identify, evaluate, prioritise, and control changes to the project when requested by a team member or stakeholder.
  • Changes must be analysed before implementation, recorded with before-and-after details, and reported to all affected parties.
  • No change is applied silently — every modification to a baselined item goes through the formal change control process in §9.5.
  • Quality and consistency are preserved throughout the change process; records and revised plans are maintained when scope or deliverables change significantly.

Guidelines that support SCM objectives

When SCM objectives require changes to the project plan or product, the following guidelines ensure objectives are met in practice:

  • Transparency: No one is allowed to change anything in the project without others knowing — all changes are visible to the team.
  • Quality preservation: Every change must ensure that overall product quality is maintained or improved, not degraded.
  • Formal change control system: All changes follow the documented change control procedure — not informal agreements.
  • Flexibility: The project process must be flexible enough that legitimate changes can be accommodated without excessive delay.
  • Records and revised plans: Records are properly maintained after each change, and the revised project plan reflects the plan actually in use by the team.
  • Projects that need to manage changes to project scope or product description must have a configuration management plan at the outset — see §9.4.

SCM objectives in project management context

  • Configuration management is part of the overall change management approach in SPM — see §3.2 Project Management Process.
  • When deliverables, schedule, budget, or approach change significantly, the project plan is revised at the end of each major lifecycle phase and approved by senior management.
  • SCM objectives support risk management by detecting and correcting configuration problems early before they propagate across the system.
  • SCM integrates with continuous integration and deployment (CI/CD) to automate testing and delivery of controlled, reproducible builds.
  • SCM is a key process area at CMM Level 2 — formal SCM procedures are required for repeatable project management — see §3.10 CMM.
  • SQA ensures adequate change management practices are instituted — linking SCM objectives to quality assurance in §8.2 SQA.
Exam question — Objectives of SCM

Q: List the objectives of Software Configuration Management. Distinguish between classic CM objectives, system-level objectives, and operational standards objectives.

Model answer points:

  • SCM = umbrella activity; manages/tracks product; controls software/hardware/tools; ensures stakeholder visibility.
  • Classic 3: identify configuration at points in time; systematically control changes; maintain integrity & traceability.
  • System-level: control evolution, collaboration, version control, replication/distribution, stakeholder visibility, reduce vulnerability.
  • Operational standards: remote admin, reduced downtime (interchangeable systems), reliable backups, easy setup, multi-user support, uniform remote software installation.
  • Change control objective: identify, evaluate, prioritise, control changes; no silent changes; quality preserved.
  • Guidelines: transparency, quality, formal system, flexibility, records + revised plan; configuration management plan required for scope/product changes.
  • PM context: part of change management; plan revision at phase end; supports CI/CD, risk management, CMM Level 2, SQA.
