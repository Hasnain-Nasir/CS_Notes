9.4 SCM Activities

← Topic 9

SCM activities are the specific tasks performed throughout a project to implement configuration management. They provide a disciplined environment for controlling work products and ensure that every change is identified, evaluated, approved, implemented, verified, and recorded.

Primary reasons for implementing SCM

  • Multiple people work on software that is continually being updated across versions, branches, and geographic locations.
  • Changes in user requirements, policy, budget, and schedule must be accommodated without destabilising the product.
  • Software must run on various machines and operating systems with reproducible builds.
  • SCM develops coordination among stakeholders and controls the costs of making changes.

The five SCM activities

Configuration management involves five interrelated activities that together form the complete SCM process:

1. Configuration identification and establishment

  • Configuration identification determines the scope of the software system — selecting which work products become configuration items.
  • It establishes relationships among CIs and creates a mechanism to manage multiple levels of control.
  • Baselines are formed at given points in time — a set of mutually consistent CIs that have been formally reviewed and agreed upon.
  • Once identified, even items not yet fully understood can be placed under control — you can manage something before you know all its details.
  • This activity also defines the change management procedure that will govern all subsequent modifications.

2. Version control

  • Version control creates and manages versions and specifications of existing products to build new products.
  • When a configuration object changes — for example from version 1.0 to 1.1 — minor corrections produce 1.1.1 and 1.1.2, followed by major updates such as 1.2 or 2.0.
  • Multiple evolutionary paths (e.g. version 1.x and 2.x) can be supported simultaneously for different releases or customers.
  • Full detail is covered in §9.6 Software Version Control.

3. Change control

  • Change control is a procedural method that ensures quality and consistency when changes are made to configuration objects.
  • A change request is submitted, evaluated, and decided upon by the Change Control Board (CCB).
  • Approved changes produce an Engineering Change Request (ECR); the CI is checked out, modified, tested, and checked back in.
  • Full process detail is in §9.5 Change Control Process.

4. Configuration status accounting

  • Configuration status accounting tracks each release during the SCM process.
  • It records what each version contains and the changes that led to that version.
  • Status accounting provides an audit trail of who changed what, when, and why — supporting accountability and reporting.
  • It enables the team to answer questions such as "What is the current approved configuration?" and "What changed between version 1.2 and 1.3?"

5. Configuration auditing and reporting

  • Configuration auditing verifies that all software products satisfy baseline requirements — it confirms that what is built is what was specified.
  • Audits complement formal technical reviews by focusing on the technical correctness of modified configuration objects.
  • Audits confirm completeness, correctness, and consistency of items in the SCM system and track action items to closure.
  • Reporting provides accurate status and configuration data to developers, testers, end users, customers, and stakeholders through release notes, installation guides, configuration guides, and memos.
scm_activity_flow.pseudo 
SCM_PROCESS(project)

ACTIVITY 1 — CONFIGURATION IDENTIFICATION
  SELECT work products to become Configuration Items (CIs)
  DEFINE relationships among CIs
  ESTABLISH baselines at milestones (formally reviewed & agreed)
  DEFINE change management procedure

ACTIVITY 2 — VERSION CONTROL
  ASSIGN version numbers to CIs (e.g. 1.0 → 1.1 → 1.2 → 2.0)
  STORE prior versions in repository
  SUPPORT multiple evolutionary paths (branches/releases)

ACTIVITY 3 — CHANGE CONTROL
  RECEIVE Change Request (CR)
  EVALUATE technical merit, side effects, impact, cost
  SUBMIT change report to Change Control Board (CCB)
  IF approved THEN
    GENERATE Engineering Change Request (ECR)
    CHECK OUT CI → MODIFY → TEST → CHECK IN
    CREATE new version
  ELSE
    NOTIFY requestor with reason for rejection
  END IF

ACTIVITY 4 — STATUS ACCOUNTING
  RECORD what each version contains
  TRACK who changed what, when, and why
  REPORT current configuration status

ACTIVITY 5 — CONFIGURATION AUDIT & REPORTING
  VERIFY product satisfies baseline requirements
  CONFIRM completeness, correctness, consistency
  TRACK audit action items to closure
  PUBLISH status to stakeholders (release notes, guides)

END SCM_PROCESS
Exam question — SCM Activities

Q: List and explain the main SCM activities. What is the role of baselines and configuration auditing?

Model answer points:

  • Five activities: (1) configuration identification/establishment, (2) version control, (3) change control, (4) status accounting, (5) configuration audit & reporting.
  • Identification: select CIs, define relationships, establish baselines, define change procedure.
  • Version control: manage versions/branches; multiple evolutionary paths supported.
  • Change control: CR → evaluation → CCB → ECR → check-out/modify/test/check-in.
  • Status accounting: track releases, record changes, audit trail.
  • Audit: verify baseline satisfaction; confirm completeness/correctness; reporting to stakeholders.
  • Baseline = formally accepted CI snapshot; changed only via formal change control.
