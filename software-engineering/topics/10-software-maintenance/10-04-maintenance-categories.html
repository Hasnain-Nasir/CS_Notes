10.4 Maintenance Categories

← Topic 10

IEEE and ISO/IEC 12207 classify all maintenance activities into four categories based on the purpose of the change — not based on who requested it or how urgent it is. Correct classification drives prioritisation, effort estimation, budget allocation, and change control routing — see §9.5 Change Control Process. Misclassification is a common exam trap — e.g. porting to a new OS is adaptive, not perfective.

Formal definitions (IEEE / ISO)

  • IEEE 1993: Software maintenance is modification of a software product after delivery to correct faults, improve performance or other attributes, or adapt the product to a modified environment.
  • ISO/IEC 12207: Maintenance activities occur when software undergoes modifications to code and associated documentation due to a problem or a need for improvement or adaptation.
  • All four categories fall under this umbrella — the category is determined by why the change is being made.
  • After receiving a maintenance request, the first step is to identify the category; sometimes the reported problem is with the user, not the software — see §10.1 Overview.

Four categories of maintenance

CategoryFormal purposeTriggerExamples
Corrective Correct discovered problems after delivery — faults that cause incorrect behaviour Bug report, failure in production, audit finding, security vulnerability Fixing a payroll calculation error; correcting a null-pointer crash; patching an SQL injection flaw; fixing incorrect tax computation in reports
Adaptive Keep software usable in a changed or changing environment — functionality unchanged New OS, hardware, DBMS, regulations, business rules, or external system interfaces Porting from Windows Server 2016 to 2022; updating JDBC drivers for PostgreSQL 16; modifying code for new GDPR rules; interfacing with a new payment gateway API
Perfective Improve performance, maintainability, or user-visible functionality after delivery User enhancement request, performance bottleneck, usability improvement, restructuring for changeability Adding export-to-PDF feature; reducing search response time from 5s to 0.5s; redesigning modules for easier modification; improving dashboard UI layout
Preventive Detect and correct latent faults before they become effective faults; reduce future maintenance difficulty Proactive quality initiative, technical debt reduction, scheduled review Refactoring high-complexity modules; updating outdated design documents; adding automated backups; improving test coverage from 40% to 80%; adding monitoring and alerting

Corrective maintenance — detail

  • Corrective maintenance is triggered when the system behaves incorrectly — output is wrong, the system crashes, or a security flaw is discovered.
  • It may be essential either to rectify bugs observed while the system is in use, or in some cases to enhance performance where the current behaviour is technically incorrect (e.g. slow response that violates the SLA).
  • Corrective changes follow the full change control process: problem report → analysis → fix → test → deploy — though critical fixes may use an expedited path.
  • Retesting verifies that the specific defect fix works; regression testing ensures the fix did not break existing functionality — both are mandatory after corrective changes.
  • Corrective maintenance is typically the least expensive category per individual change, because the scope is narrow and the problem is well-defined.

Adaptive maintenance — detail

  • Adaptive maintenance modifies the product so it continues to work when the environment changes — the specified functional requirements remain the same.
  • Common triggers include new operating systems, new hardware platforms, new database versions, changed government policies, revised business rules, and new external system interfaces.
  • Adaptive work is often underestimated — porting may require changes throughout the codebase even when user-visible features are unchanged.
  • Deferred adaptive maintenance leaves the system on unsupported platforms with security vulnerabilities and compliance risks.
  • Example: a banking system must adapt when the central bank changes transaction reporting format — functionality is the same (report transactions) but the format changes.

Perfective maintenance — detail

  • Perfective maintenance is driven by user demands for new features or improvements to existing functionality — it is the largest category by effort.
  • It includes improving functionality, performance, reliability, and restructuring the system to improve changeability (maintainability).
  • Perfective changes expand scope compared to corrective fixes — they require requirements analysis, design updates, new test cases, and user documentation revisions.
  • Because perfective requests are open-ended, they need strong change control and prioritisation to prevent scope creep — see §9.5.
  • Example: users request a mobile-responsive version of an existing web dashboard — this is perfective (new capability), not adaptive (environment change).

Preventive maintenance — detail

  • Preventive maintenance addresses problems that are not yet critical but may cause serious failures if ignored — it reduces future corrective effort.
  • It includes code optimisation, documentation updates, scheduled reviews, test suite expansion, backup procedures, and refactoring to reduce complexity.
  • Preventive work is closely related to proactive maintenance — taking action before failures occur rather than reacting after them.
  • In ISO terminology, preventive maintenance detects and corrects latent faults — defects present in the code that have not yet caused a visible failure.
  • Preventive maintenance reduces the C (complexity) term in the maintenance effort formula — see §10.5 Cost of Maintenance.

Proactive vs reactive maintenance

DimensionProactiveReactive
TimingScheduled before problems occurTriggered after a failure or complaint
Primary categoryPreventive (and planned perfective/adaptive)Corrective (and emergency adaptive)
ExamplesScheduled refactoring, documentation review, backup testingHotfix for production crash, emergency OS port after vendor EOL
Cost patternLower long-term cost; predictable budgetHigher long-term cost; unpredictable spikes
  • Planned maintenance activities (scheduled updates, backups, reviews) are proactive; unplanned activities triggered by crashes or security breaches are reactive.
  • Organisations with strong preventive programmes spend less on emergency corrective work over the system lifetime.

Corrective maintenance and patching

  • Patching is emergency corrective maintenance — a fix applied under management pressure when a critical production defect cannot wait for the next scheduled release.
  • Patches are deployed quickly but carry higher risk because impact analysis, full regression testing, and documentation updates may be abbreviated.
  • Poorly managed patches often introduce unforeseen future errors — a patch in one module breaks another module that was not retested.
  • Even emergency patches must not bypass change control entirely — at minimum: unique change number, impact analysis, retest of affected areas, and post-deployment monitoring.
  • Every patch should be merged back into the main codebase through version control — see §9.6 Version Control — to prevent patch divergence.

Typical effort and cost distribution

  • Industry studies consistently show that maintenance effort is dominated by perfective work, not bug fixing — this is why maintenance budgets are often underestimated.
  • Approximate distribution: ~60% perfective, ~20% adaptive, ~20% corrective.
  • Preventive effort is often embedded within the other categories rather than tracked separately — e.g. refactoring done during a perfective change.
  • Corrective maintenance is typically less expensive per change than perfective maintenance because scope is narrower and requirements are clearer.
  • Organisations that budget maintenance only for bug fixes allocate roughly one-third of what they actually need.

Classify and route maintenance request

classify_maintenance_request.pseudo 
FUNCTION classify_and_route(request):

  // Step 1 — Validate: is it actually a software problem?
  IF request.problem_caused_by_user_error THEN
    RETURN "user_training_required"   // not a maintenance change
  END IF

  // Step 2 — Classify by purpose
  IF request.trigger IN ["production_failure", "bug_report", "security_flaw"] THEN
    category = "corrective"
  ELSE IF request.trigger == "environment_change" THEN
    // new OS, hardware, DBMS, regulations — functionality unchanged
    category = "adaptive"
  ELSE IF request.trigger IN ["user_enhancement", "performance_improvement",
                              "new_feature", "ui_improvement"] THEN
    category = "perfective"
  ELSE IF request.trigger IN ["proactive_refactoring", "documentation_update",
                              "backup_setup", "test_coverage_improvement"] THEN
    category = "preventive"
  ELSE
    category = "unclassified"   // requires manual review by maintainer
  END IF

  // Step 3 — Assign change number and enter repository (SCM)
  change_id = assign_unique_change_number(request)
  store_in_modification_request_repository(change_id, category, request)

  // Step 4 — Route by priority
  IF category == "corrective" AND request.priority == "critical" THEN
    ROUTE to emergency_patch_process
    REQUIRE impact_analysis BEFORE deploy
    REQUIRE retest + regression_test AFTER fix
  ELSE IF category == "perfective" THEN
    ROUTE to change_control_board for scope/priority approval
  ELSE
    ROUTE to standard_maintenance_process
  END IF

  RETURN category, change_id
END FUNCTION

// Example classifications:
// "App crashes on login"           → corrective (critical patch)
// "Migrate to Linux from Windows"  → adaptive
// "Add dark mode to UI"            → perfective
// "Refactor payment module"        → preventive
Exam question — Maintenance Categories

Q: Define the four categories of software maintenance with formal definitions and examples. Distinguish proactive from reactive maintenance. State effort distribution. Classify given scenarios. Solve UGC NET 2016 match question.

Model answer points:

  • IEEE: modification after delivery to correct faults, improve attributes, or adapt to modified environment.
  • Corrective: fix discovered problems/bugs; least expensive per change; retest + regression test required.
  • Adaptive: adapt to changed environment (OS, hardware, regulations) — functionality unchanged.
  • Perfective: improve performance/maintainability or add features — largest share (~60%) of effort.
  • Preventive: correct latent faults before they become effective; reduces complexity (C in effort formula).
  • Proactive = scheduled preventive/planned; Reactive = after failure (corrective/emergency adaptive).
  • Effort: ~60% perfective, ~20% adaptive, ~20% corrective; preventive embedded in others.
  • Patching = emergency corrective; needs impact analysis even when urgent; risk of introducing new bugs.
  • UGC NET 2016: i–b, ii–c, iii–d, iv–a → Answer B.
  • Common trap: OS port = adaptive (not perfective); new feature = perfective (not corrective).
