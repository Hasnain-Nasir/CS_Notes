10.1 Software Maintenance Overview

← Topic 10

Software maintenance is the phase of the software life cycle that begins after delivery and continues for as long as the system remains in use. It encompasses all activities required to keep software operational, correct, and aligned with evolving user and environmental needs — see §2.1 SDLC for the maintenance phase within the overall life cycle.

Definitions

  • IEEE Std 1219 (1993): Software maintenance is the modification of a software product after delivery to correct faults, to improve performance or other attributes, or to adapt the product to a modified environment.
  • ISO/IEC 12207: Maintenance comprises the activities that occur when software code or associated documentation is modified due to a problem or because of the need for improvement or adaptation.
  • Maintenance is a continuous process throughout the software life cycle — not merely a post-delivery activity that begins only after deployment.
  • Every change during maintenance must follow formal change control — see §9.5 Change Control Process.

Key aspects of maintenance

  • Bug fixing (corrective maintenance): Identifying and correcting defects discovered in production use — see §10.4 Maintenance Categories.
  • Enhancements (perfective maintenance): Adding new features or improving existing functionality to meet evolving user requirements.
  • Performance optimization: Tuning the system to improve response time, throughput, or resource consumption.
  • Porting and migration (adaptive maintenance): Adapting the software to new hardware platforms, operating systems, or network environments.
  • Re-engineering: Restructuring legacy systems to improve maintainability without changing external behaviour — see §10.6 Software Re-engineering.
  • Documentation: Updating user manuals, design documents, and inline comments so they reflect the current system state.

Maintenance in the SDLC

  • The SDLC explicitly includes a maintenance phase after deployment — maintenance consumes the majority of total life cycle effort and cost.
  • Maintenance activities reuse the same engineering disciplines as development: analysis, design, implementation, and testing.
  • Effective maintenance depends on configuration management, version control, and regression testing established during development — see Topic 9 — Processes & Configuration Management.

Planned vs unplanned maintenance

  • Planned maintenance: Scheduled activities such as periodic reviews, preventive refactoring, documentation updates, and scheduled upgrades.
  • Unplanned maintenance: Reactive work triggered by unexpected failures, critical bugs, or urgent regulatory changes.
  • Proactive maintenance: Anticipating future problems through code reviews, refactoring, and preventive measures before failures occur.
  • Reactive maintenance: Responding to problems only after they have been reported or have caused service disruption.
  • A mature maintenance organisation balances proactive and reactive work — over-reliance on reactive maintenance increases cost and risk.

Maintenance stakeholders

  • Original development team: Maintains the system they built; has the deepest knowledge but may move to new projects after delivery.
  • In-house maintenance team: A dedicated group within the organisation that takes over after the development team hands off the product.
  • Third-party maintenance: External vendors or contractors maintain the system under a support contract — common when the original developers are unavailable.
  • Knowledge transfer from developers to maintainers is critical — poor handover leads to longer diagnosis times and higher maintenance cost.

Maintenance process phases (ISO/IEC 12207)

The ISO maintenance process defines seven sequential phases, each with defined inputs, activities, and outputs.

PhaseInputsProcess / ActivitiesOutputs
1. Problem Identification User reports, monitoring alerts, change requests, audit findings Log and classify maintenance requests; assign priority; determine if change is needed Maintenance request record; problem classification
2. Analysis Maintenance request; current system documentation; source code Investigate root cause; perform impact analysis; estimate effort and cost; propose solution Analysis report; impact assessment; feasibility decision
3. Design Approved analysis report; existing design documents Design the modification; update affected modules; plan test cases; obtain design review approval Updated design specification; test plan
4. Implementation Approved design; source code baseline; change control authorisation Modify code and documentation; conduct code review; check in changes via SCM Modified software; updated documentation; change log entry
5. System Test Modified software; test plan; test environment Execute unit, integration, and system tests; perform regression testing; verify fix or enhancement Test results; defect reports; test completion report
6. Acceptance Test Tested software; acceptance criteria; user involvement User or customer validates that the change meets requirements; sign-off obtained Acceptance test report; user approval
7. Delivery Accepted software; release notes; deployment plan Deploy to production; update configuration baselines; notify stakeholders; close maintenance request Released software; updated baselines; maintenance closure record

Maintenance request flow

maintenance_request_flow.pseudo 
// ISO 12207 maintenance process — simplified flow

REQUEST = receive_maintenance_request(user_report OR change_request)
LOG request in maintenance tracking system

CLASSIFY request AS corrective | adaptive | perfective | preventive
ASSIGN priority (critical | high | medium | low)

IF priority == critical THEN
  escalate to emergency patch process
ELSE
  submit to change control board   // see §9.5
END IF

ANALYSIS = perform_impact_analysis(REQUEST, current_baseline)
IF ANALYSIS.feasible == false THEN
  reject request WITH reason
  GOTO close_request
END IF

DESIGN modification based on ANALYSIS
IMPLEMENT changes under SCM version control
TEST modified system (unit → integration → system → regression)
ACCEPTANCE_TEST with user/customer sign-off
DELIVER to production; update baselines; close request
Exam question — Software Maintenance Overview

Q: Define software maintenance (IEEE and ISO). List key maintenance aspects. Describe the ISO maintenance process phases. Differentiate planned vs unplanned and proactive vs reactive maintenance.

Model answer points:

  • IEEE: modification after delivery to correct faults, improve performance/attributes, adapt to modified environment.
  • ISO/IEC 12207: activities when code/docs modified due to problem or improvement/adaptation need.
  • Continuous throughout lifecycle; not just post-delivery; part of SDLC maintenance phase.
  • Key aspects: bug fixing, enhancements, performance, porting/migration, re-engineering, documentation.
  • Seven phases: Problem Identification → Analysis → Design → Implementation → System Test → Acceptance Test → Delivery.
  • Planned = scheduled; unplanned = reactive to failures; proactive = preventive; reactive = fix after failure.
  • Stakeholders: original dev team, in-house maintenance, third-party; all changes via change control (§9.5).
