10.2 Software as Evolutionary Entity

← Topic 10

Software is not a static artifact — it is a dynamic model of the real world that must evolve as the world itself changes. This evolutionary nature is the fundamental reason maintenance exists and why it dominates the software life cycle — see §10.3 Need for Maintenance and §10.4 Maintenance Categories.

Software as a model of the real world

  • Software encodes a simplified representation of business processes, physical systems, or user workflows.
  • When the real world changes — new regulations, market conditions, technology, or user expectations — the software model must be updated to remain valid.
  • A system that is never modified eventually becomes obsolete or incorrect relative to its operating environment.
  • Maintenance is therefore not an optional add-on but an inherent consequence of software's role as a world model.

E-type vs S-type software

  • S-type (Static-type) software: Software that implements a well-understood, fixed specification — e.g. scientific computations with stable mathematical foundations; changes are rare and predictable.
  • E-type (Evolutionary-type) software: Software that implements a real-world application and must evolve as the application domain changes — virtually all business and information systems are E-type.
  • E-type software is subject to Lehman's laws of software evolution; S-type software is largely exempt from them.
  • Most exam questions on software evolution refer to E-type systems.

Lehman's eight laws of software evolution

Meir Lehman (1980) formulated eight empirically observed laws governing E-type software evolution.

  1. Continuing Change: An E-type system must be continually adapted or it becomes progressively less useful over time.
  2. Increasing Complexity: As a system evolves, its structure becomes more complex unless active work is done to reduce complexity.
  3. Self Regulation: The evolution process is self-regulating — distribution of product and process measures tends to be near-normal.
  4. Conservation of Organisational Stability: The average effective global activity rate in an evolving E-type system is invariant over the product lifetime.
  5. Conservation of Familiarity: During active evolution, the content of successive releases is statistically invariant — roughly the same proportion of the system is changed in each release.
  6. Continuing Growth: The functional content of an E-type system must be continually increased to maintain user satisfaction over its lifetime.
  7. Declining Quality: The quality of an E-type system will appear to be declining unless it is rigorously maintained and adapted to operational environment changes.
  8. Feedback System: E-type systems constitute multi-loop, multi-agent feedback systems and must be treated as such to achieve significant improvement.

Software is never "finished"

  • Unlike manufactured products with a defined end-of-life, software systems can remain in active use for decades.
  • Typical development takes 1–2 years, while maintenance may continue for 15–20 years or longer.
  • More than 65% of total life cycle cost is spent on maintenance — see §10.5 Cost of Maintenance.
  • Organisations that treat delivery as "project completion" underestimate the long-term commitment required.

Legacy systems

  • A legacy system is an older software system that remains in operational use despite being built with outdated technology, languages, or architectures.
  • Legacy systems often lack adequate documentation, have high maintenance cost, and resist integration with modern platforms.
  • Organisations face a strategic choice: continue maintaining, re-engineer, replace, or retire legacy systems — see §10.6 Re-engineering.
  • Reverse engineering is often the first step in understanding a legacy system before deciding on a migration strategy — see §10.7 Reverse Engineering.
  • Many critical business systems (banking, government, healthcare) are legacy systems that cannot be simply discarded.
Exam question — Software as Evolutionary Entity

Q: Explain why software is an evolutionary entity. State Lehman's eight laws. Differentiate E-type and S-type software. What is a legacy system?

Model answer points:

  • Software models the real world; real-world changes require software changes; maintenance is inherent not optional.
  • E-type = real-world application, evolves with domain; S-type = fixed specification, stable, rare changes.
  • Lehman's 8 laws: Continuing Change, Increasing Complexity, Self Regulation, Conservation of Organisational Stability, Conservation of Familiarity, Continuing Growth, Declining Quality, Feedback System.
  • Software never finished: build 1–2 years, maintain 15–20 years; >65% of lifecycle cost in maintenance.
  • Legacy system = old operational software with outdated tech, poor docs, high cost; options: maintain, re-engineer, replace, retire.
