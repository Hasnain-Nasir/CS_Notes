Key takeaways — exam revision

← Topic 10
  • 10.1 Overview: IEEE: modification after delivery to correct faults, improve performance/attributes, adapt to modified environment. ISO/IEC 12207: activities when code/docs modified due to problem or improvement/adaptation. Continuous throughout lifecycle; part of SDLC maintenance phase (§2.1).
  • Key aspects: Bug fixing, enhancements, performance optimization, porting/migration, re-engineering, documentation. Planned vs unplanned; proactive vs reactive. Stakeholders: original dev team, in-house maintenance, third-party.
  • ISO maintenance process (7 phases): Problem Identification → Analysis → Design → Implementation → System Test → Acceptance Test → Delivery. All changes via change control (§9.5).
  • 10.2 Evolutionary entity: Software models real world; real-world changes require software changes. E-type = evolutionary (business systems); S-type = static (fixed spec). Software never finished: build 1–2 years, maintain 15–20 years.
  • Lehman's 8 laws: Continuing Change, Increasing Complexity, Self Regulation, Conservation of Organisational Stability, Conservation of Familiarity, Continuing Growth, Declining Quality, Feedback System.
  • Legacy systems: Old operational software with outdated tech, poor docs, high cost; options: maintain, re-engineer, replace, retire.
  • 10.3 Need: Correct faults, improve design, enhancements, interface with other systems, new hardware/OS, migrate legacy, retire software, user requirement changes. Challenges: 10–15 year lifespan, undocumented changes, legacy code, complexity, changing requirements, interoperability, lack of tests, skill gaps, high cost.
  • Support processes: Maintenance plan, SCM (Topic 9), change control (§9.5), regression testing (Topic 7), current documentation.
  • 10.4 Categories (IMPORTANT): IEEE four categories by purpose. Corrective = fix discovered problems (least expensive per change; retest + regression). Adaptive = changed environment, same function (OS port, regulations). Perfective = improve performance/features/restructure (~60% effort). Preventive = correct latent faults before failure; reduces C in effort formula.
  • Proactive vs reactive: Proactive = scheduled preventive/planned; Reactive = after failure (corrective/emergency). Patching = emergency corrective; needs impact analysis; risk of new bugs.
  • Exam traps: OS port = adaptive (NOT perfective); new feature = perfective (NOT corrective); refactor = preventive. UGC NET 2016: i–b, ii–c, iii–d, iv–a → Answer B. Memory: C=Correct bugs, A=Adapt environment, P=Perfect/improve, P=Prevent latent faults.
  • 10.5 Cost (IMPORTANT): >65% lifecycle cost; build 1–2 yrs, maintain 15–20 yrs. M = P + Ke(C−D): legacy example P=50,Ke=6,C=8,D=3 → M=80 PM; after familiarity D=6 → M=62 PM; after refactoring C=5 → M=44 PM.
  • COCOMO maintenance: AME = ACT × SDE. Example: 100 KLOC organic → SDE≈302 PM; ACT=0.15 → AME=45.3 PM/yr; breakeven ~6.7 years. Corrective cheaper per change than perfective.
  • 10 cost factors: Complexity, size, users, change rate, personnel, tools, plan, age, maintenance type, location. Non-technical: domain, staff stability, lifetime, external dependence, hardware stability. Technical: module independence, language, style, testing, docs, SCM.
  • Postponing maintenance: Customer dissatisfaction, hidden errors, quality decline, technical debt, security exposure, staff turnover (reduces D). Reduce cost: design for maintainability, proactive strategy, automation, Agile/DevOps, SCM, re-engineering.
  • 10.6 Re-engineering: Examine and alter system to reconstitute in new form; improve maintainability; reuse existing artifacts. Not new development; not reuse (§12.1). Process: Planning → Analysis → Design → Implementation → Testing → Deployment.
  • Six re-engineering steps: Inventory Analysis → Document Reconstruction → Reverse Engineering → Code Reconstruction → Data Reconstruction → Forward Engineering. Cost factors: software quality, tool support, data conversion, expert staff.
  • 10.7 Reverse engineering: Recover design/requirements from code; back engineering; does not modify software. Why: lost docs, maintenance, reuse, discover flaws, security analysis.
  • 8 reverse engineering steps: Collect info → Examine → Extract structure → Record functionality → Record data flow → Record control flow → Review design → Generate documentation.
  • Forward vs reverse: Forward = requirements→design→code (creates software); Reverse = code→design→requirements (produces documentation). Data RE: program level + system level. Tools: Rigi, Bunch, GEN++, PBS, CIAO.
