10.3 Need for Maintenance

← Topic 10

Maintenance is required whenever software must remain correct, useful, and compatible with its operating environment. The reasons span technical defects, evolving requirements, and environmental change — each maps to one of the four maintenance categories described in §10.4 Maintenance Categories.

Primary reasons for maintenance

  • Correct faults: Fix bugs and errors discovered during production use that were not caught during testing.
  • Improve design: Restructure internal modules to reduce complexity and improve future changeability without altering external behaviour.
  • Implement enhancements: Add new features or extend existing functionality to meet evolving business needs.
  • Interface with other systems: Modify the software to interoperate with newly introduced external systems, APIs, or data formats.

Environmental and organisational drivers

  • Accommodate new hardware/software/telecom: Adapt to new processors, operating systems, databases, network protocols, or cloud platforms.
  • Migrate legacy systems: Move data and functionality from obsolete platforms to modern infrastructure while preserving business logic.
  • Retire software: Plan and execute graceful decommissioning, data archival, and user migration to replacement systems.
  • User requirement changes: Users discover new needs after deployment that were not captured in the original requirements specification.
  • Regulatory and compliance changes: New laws or industry standards may require software modifications to remain compliant.

Challenges of software maintenance

  • Long lifespan (10–15 years): Systems outlive the developers who built them, creating knowledge gaps.
  • Undocumented changes: Ad-hoc fixes applied without updating design documents accumulate technical debt over time.
  • Legacy code: Old code written in obsolete languages with poor structure resists modification and testing.
  • Complexity growth: Lehman's law of increasing complexity means each change makes the next change harder — see §10.2.
  • Changing requirements: Users continuously request new features, making it difficult to stabilise the system.
  • Interoperability: Integrating with new external systems requires understanding both the legacy system and the new interface standards.
  • Lack of test coverage: Legacy systems often have no automated regression test suite, making every change risky.
  • Skill gaps: Maintainers may lack expertise in the original programming language, framework, or domain.
  • High cost: Maintenance consumes the majority of life cycle budget — see §10.5 Cost of Maintenance.

Importance of planning and supporting processes

  • A formal maintenance plan should be prepared before delivery, specifying staffing, procedures, SLAs, and escalation paths.
  • Software Configuration Management (SCM) ensures every change is tracked, baselined, and reversible — see Topic 9 §9.1 SCM Overview.
  • Change control prevents unauthorised modifications and ensures impact analysis before implementation — see §9.5 Change Control Process.
  • Testing and regression testing verify that fixes and enhancements do not introduce new defects — see Topic 7 — Software Testing & Debugging.
  • Documentation must be kept current so maintainers can understand system behaviour without relying on original developers.
  • Design for maintainability during development — modular architecture, clear interfaces, and coding standards reduce future maintenance cost.
Exam question — Need for Maintenance

Q: List reasons why software maintenance is needed. What challenges make maintenance difficult? How do SCM, change control, and testing support maintenance?

Model answer points:

  • Reasons: correct faults, improve design, enhancements, interface with other systems, new hardware/OS, migrate legacy, retire software, user requirement changes.
  • Challenges: 10–15 year lifespan, undocumented changes, legacy code, complexity growth, changing requirements, interoperability, lack of tests, skill gaps, high cost.
  • Support: maintenance plan before delivery; SCM for tracking/baselines (Topic 9); change control for authorised changes (§9.5); regression testing (Topic 7); current documentation.
