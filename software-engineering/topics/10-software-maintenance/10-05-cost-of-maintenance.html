10.5 Cost of Maintenance

← Topic 10

Software maintenance is the most expensive phase of the software life cycle — more than 65% of total cost is spent here, not during initial development. Understanding what drives maintenance cost, how to estimate effort using quantitative models, and how to reduce cost is essential for both project planning and exams. Maintenance category mix directly affects cost — see §10.4 Maintenance Categories.

Why maintenance cost exceeds development cost

  • More than 65% of total software life cycle cost is spent on maintenance activities — not on initial development.
  • Software is typically built in 1–2 years but maintained for 15–20 years or longer — the maintenance period is 10× longer than development.
  • Approximately 60% of maintenance effort is perfective (enhancements and improvements) — not bug fixing — so budgets based only on corrective work are severely inadequate.
  • Each maintenance change touches code that other modules depend on — ripple effects increase analysis and regression testing effort.
  • Original developers leave; new maintainers must rebuild understanding from code and incomplete documentation — familiarity (D) starts low.
  • Complexity (C) grows with every change unless preventive refactoring is performed — Lehman's Law of Increasing Complexity applies.
  • Legacy systems use obsolete languages and platforms — specialised skills are scarce and expensive.
  • Undocumented changes accumulate over years — each new maintainer spends extra effort discovering what previous teams changed.
  • Organisations that under-budget maintenance face escalating technical debt and eventual system failure or forced replacement.

Components of maintenance cost and effort

Cost componentWhat it includes
LaborSalaries of maintenance engineers, analysts, testers, project managers — the largest single cost component
Hardware and software toolsDevelopment environments, test servers, SCM tools, deployment platforms, software licences
TrainingOnboarding maintainers to unfamiliar codebases, languages, or domain knowledge lost when original team leaves
Time and resourcesEffort to identify the problem, design the fix, implement, test, and deploy the solution
Communication and coordinationMeetings with users, management, and other teams; status reporting; change approval cycles
Testing and validationRetesting the fix, regression testing unaffected areas, acceptance testing with users
  • Maintenance effort activities include: problem analysis, feasibility study, design modification, coding, unit testing, integration testing, system testing, acceptance testing, and documentation update.
  • Indirect costs include production downtime during failed deployments, customer support for maintenance incidents, and opportunity cost of staff not available for new product development.

Maintenance effort formula (Boehm)

  • Boehm's model relates total maintenance effort to productive baseline effort, system complexity, and team familiarity:

M = P + Ke(C − D)

SymbolMeaningEffect on M
MTotal maintenance effort (person-months or person-hours)Output — what we estimate
PProductive effort — baseline effort for a well-designed, well-documented system with experienced teamFixed baseline; lower P = better initial quality
KeEmpirical constant reflecting organisation and environment (typically 4–8)Higher Ke = steeper penalty for complexity
CComplexity measure — increases with poor design, high coupling, lack of documentation, large codebaseHigher C → higher M
DFamiliarity measure — increases as maintainers gain knowledge of the system over timeHigher D → lower M
  • When C > D, the term Ke(C − D) adds penalty effort — legacy systems with poor docs have high C and low D, maximising this penalty.
  • As maintainers gain experience over months/years, D increases and M decreases for similar types of changes.
  • Preventive maintenance (refactoring, documentation) reduces C — directly lowering future M.
  • Staff turnover resets D — each new maintainer starts with low familiarity, increasing cost.

Worked examples — Boehm formula

boehm_effort_examples.pseudo 
// Formula: M = P + Ke * (C - D)

// Example 1 — Well-maintained system, experienced team
P = 40,  Ke = 5,  C = 4,  D = 3
M = 40 + 5 * (4 - 3) = 40 + 5 = 45 person-months

// Example 2 — Legacy system, poor docs, new team (EXAM FAVOURITE)
P = 50,  Ke = 6,  C = 8,  D = 3
M = 50 + 6 * (8 - 3) = 50 + 30 = 80 person-months

// Example 3 — Same legacy system after 2 years (team gained familiarity)
P = 50,  Ke = 6,  C = 8,  D = 6
M = 50 + 6 * (8 - 6) = 50 + 12 = 62 person-months

// Example 4 — After preventive refactoring (complexity reduced)
P = 50,  Ke = 6,  C = 5,  D = 6
M = 50 + 6 * (5 - 6) = 50 + (-6) = 44 person-months
// C < D → penalty term is negative → M < P (ideal state)

COCOMO maintenance effort estimation

  • COCOMO provides a top-down estimate of annual maintenance effort based on original development size:
  • AME = ACT × SDE
  • AME = Annual Maintenance Effort (person-months per year)
  • ACT (Annual Change Traffic): The fraction of the software modified per year — e.g. ACT = 0.15 means 15% of the codebase changes annually.
  • SDE (Software Development Effort): The original total effort in person-months to develop the software — obtained from Basic COCOMO: E = a × (KLOC)b.
  • See §3.6 COCOMO Model for development effort formulas and project type constants.

Full COCOMO maintenance example

cocomo_maintenance_calculation.pseudo 
// Step 1 — Estimate original development effort (Basic COCOMO, Organic)
KLOC = 100
a = 2.4,  b = 1.05
SDE = a * (KLOC ^ b)
// SDE = 2.4 * (100 ^ 1.05) = 2.4 * 125.9 ≈ 302 person-months

// Step 2 — Annual Maintenance Effort
ACT = 0.15    // 15% of codebase changed per year
AME = ACT * SDE
// AME = 0.15 * 302 = 45.3 person-months per year

// Step 3 — Multi-year maintenance projection
maintenance_years = 10
total_maintenance = AME * maintenance_years
// total = 45.3 * 10 = 453 person-months over 10 years

// Step 4 — Compare to development
// Development: 302 PM (one-time)
// 10-year maintenance: 453 PM
// Maintenance exceeds development cost in ~6.7 years:
// breakeven_years = SDE / AME = 302 / 45.3 ≈ 6.7 years

// Step 5 — Boehm check on a specific change cycle
P = 45,  Ke = 6,  C = 7,  D = 4
M = P + Ke * (C - D) = 45 + 6 * 3 = 63 person-months for this cycle

3Number of usersMore users → more change requests, higher impact of failures, more testing scenarios
4Change rateMore frequent changes → more cumulative complexity and regression risk

Risks and intangible costs of postponing maintenance

  • Organisations sometimes try to control maintenance cost by postponing maintenance — this creates intangible costs that exceed the savings.
  • Customer dissatisfaction: Unresolved bugs and unfulfilled enhancement requests erode trust — users may switch to competitors or escalate contractually.
  • Hidden errors: Deferred fixes allow latent defects to compound — a minor bug today may cause a critical production failure months later.
  • Reduced software quality: Changes made under pressure without proper testing introduce new hidden errors — quality degrades with each deferred cycle.
  • Technical debt: Deferred preventive and perfective work accumulates — eventually the system becomes unmaintainable and requires expensive re-engineering or replacement.
  • Security exposure: Postponed adaptive maintenance (OS upgrades, library patches) leaves known vulnerabilities unpatched.
  • Staff morale: Maintainers working on fragile, undocumented systems with constant firefighting experience higher burnout and turnover — further reducing D.

Ways to reduce maintenance cost

  • Design for maintainability during development: Modular architecture, coding standards, comprehensive documentation, and automated tests reduce C from the start — cheapest time to invest in quality is during initial development.
  • Proactive maintenance strategy: Schedule preventive refactoring and documentation updates rather than waiting for failures — reduces emergency corrective spikes.
  • Automation: Automated regression test suites, CI/CD pipelines, and deployment scripts reduce manual effort per change.
  • Agile, DevOps, and CI/CD: Short release cycles with continuous integration catch defects early when they are cheap to fix.
  • SCM tools: Version control, baselines, and change tracking prevent conflicting modifications — see §9.6 Version Control.
  • Clear maintenance plan: Define roles, SLAs, escalation procedures, and budget before delivery — not after the first production incident.
  • Retain domain knowledge: Minimise staff turnover; document decisions and changes continuously to maintain high D.
  • Re-engineering: Restructuring legacy systems to reduce complexity can lower long-term cost — see §10.6 Re-engineering.
Exam question — Cost of Maintenance

Q: Why does maintenance cost exceed development cost? Explain M = P + Ke(C−D) with a numerical example. Calculate AME using COCOMO. List technical and non-technical cost factors. State risks of postponing maintenance.

Model answer points:

  • >65% lifecycle cost in maintenance; build 1–2 years, maintain 15–20 years; ~60% perfective effort.
  • Cost components: labor (largest), tools, training, testing, coordination.
  • M = P + Ke(C−D): M=total, P=productive baseline, Ke=4–8, C=complexity, D=familiarity.
  • Legacy example: P=50, Ke=6, C=8, D=3 → M=80 PM; after familiarity D=6 → M=62 PM.
  • COCOMO: AME = ACT × SDE; e.g. SDE=302 PM, ACT=0.15 → AME=45.3 PM/year; breakeven ~6.7 years.
  • 10 cost factors: complexity, size, users, change rate, personnel, tools, plan, age, type, location.
  • Non-technical: domain, staff stability, lifetime, external dependence, hardware stability.
  • Technical: module independence, language, style, testing, documentation, SCM.
  • Corrective cheaper per change than perfective; preventive reduces C.
  • Postponing risks: customer dissatisfaction, hidden errors, quality decline, technical debt, security exposure.
  • Reduce cost: design for maintainability, proactive strategy, automation, Agile/DevOps, SCM, re-engineering.
