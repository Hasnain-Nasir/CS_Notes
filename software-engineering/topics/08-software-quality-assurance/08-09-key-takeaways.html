Key takeaways — exam revision

← Topic 8
  • 8.1 QA Overview: Evolution: Inspection → QC → QA → TQM. QA = process-oriented, preventive, proactive; QC = product-oriented, corrective, reactive. QA precedes QC. Verification = QA example ("building product right"); Validation = QC example ("building right product").
  • QA vs QC table: QA focuses on how product is built (standards, audits, reviews); QC focuses on finished product (testing, inspection, defect fixing). Both needed — QA reduces defects reaching QC.
  • 8.2 SQA: Umbrella activity parallel to entire SDLC. Elements: standards (IEEE/ISO), reviews/audits, testing oversight, defect analysis, change management, education, security, safety, risk. Focus: portability, usability, reusability, correctness, maintainability, error control.
  • SQA major activities: SQA plan, checkpoint reviews, change impact analysis, multi-level testing oversight, vendor relations, record keeping. Seven encompassed areas: standards, reviews, testing oversight, defect analysis, change management, education, security/safety/risk.
  • SQA vs testing: SQA = process assurance throughout SDLC; testing = product verification. SQA group reports to senior management independently.
  • 8.3 V&V: Verification = "Are we building the product right?" (static: reviews, inspections, walkthroughs, static analysis). Validation = "Are we building the right product?" (dynamic: testing, prototyping, UAT). Verification followed by validation at each stage. Testing detail: Topic 7.
  • V&V roles in SDLC: Traceability analysis, interface analysis, criticality analysis (5 steps: identify components → assign levels → identify failure modes → assess impact → prioritise V&V), hazard/risk analysis.
  • V-Model link: Each development phase paired with corresponding V&V activity (requirements ↔ acceptance testing, design ↔ system testing, etc.).
  • 8.4 SQA Plans: Developed during project planning; governs SQA and engineering team QA. Identifies: evaluations, audits/reviews, applicable standards, error reporting/tracking, SQA documents, feedback amount. Activities: process review, FTRs, deviation handling, senior management reporting. IEEE 730 = standard for SQA Plan contents.
  • 8.5 McCall (1977): 11 factors in 3 groups — product operation (correctness, reliability, efficiency, integrity, usability); product revision (maintainability, flexibility, testability); product transition (portability, reusability, interoperability).
  • Boehm (1978): Hierarchical model — primary uses (as-is utility, maintainability, portability) → 7 quality factors → primitive constructs (measurable attributes).
  • ISO/IEC 9126: 4 parts (quality model, external metrics, internal metrics, quality in use). Six characteristics: functionality, reliability, usability, efficiency, maintainability, portability — each with sub-characteristics. Quality in use: effectiveness, productivity, safety, satisfaction. Replaced by ISO/IEC 25010 (SQuaRE).
  • Organisational frameworks: ISO 9000 (QMS certification), CMMI (process maturity levels), Six Sigma (defect reduction).
  • 8.6 ISO 9000: International QMS standards; certifies processes, not products. Why software needs it: intangible product, client requirements, auditable framework. ISO 9001:2015 = certifiable standard. Five QMS components: QMS, management responsibility, resource management, product realization, measurement/analysis/improvement.
  • Seven principles: Customer focus, leadership, engagement of people, process approach, improvement, evidence-based decision making, relationship management. ISO/IEC/IEEE 90003:2018 = software application guide. Eight implementation steps: gap analysis → commitment → design → training → implement → internal audit → management review → external audit.
  • ISO 9000 limitations: Process not product; expensive; generic; can become paperwork; no maturity levels (unlike CMM).
  • 8.7 SEI-CMM: Strategy for process improvement (SEI, Carnegie Mellon). Five levels: 1 Initial (ad hoc) → 2 Repeatable (basic PM) → 3 Defined (standardised) → 4 Managed (quantitative) → 5 Optimizing (continuous improvement). Sequential — stabilise each before advancing.
  • CMM KPAs: L2: project planning, config mgmt, requirements mgmt, subcontract mgmt, SQA. L3: peer reviews, inter-group coordination, org process definition/focus, training. L4: software quality mgmt, quantitative process mgmt. L5: process/technology change mgmt, defect prevention. CMMI extends CMM.
  • ISO 9000 vs CMM: ISO = QMS pass/fail, all industries, document control focus. CMM = 5-level maturity rating, software-specific, quantitative management focus. Complementary — ISO asks "have a quality system?"; CMM asks "how mature is the process?"
  • 8.8 Six Sigma: Bill Smith, Motorola 1986. Target: 3.4 DPMO = 99.99966% yield. Data-driven, customer-focused, process-oriented. Six principles: customer focus, data-driven decisions, proactive management, boundaryless collaboration, drive for perfection, structured methodology.
  • DMAIC (existing process): Define → Measure → Analyze → Improve → Control. DMADV (new design): Define → Measure → Analyze → Design → Verify.
  • Six Sigma belts: White (awareness), Yellow (support), Green (part-time project lead), Black (full-time project lead), Master Black Belt (strategy/training). Tools: 5S, seven wastes (Muda), VOC, kaizen, Pareto, fishbone, control charts, FMEA. Lean Six Sigma = Six Sigma + Lean waste elimination.
