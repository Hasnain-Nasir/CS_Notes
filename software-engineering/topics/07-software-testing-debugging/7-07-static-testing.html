7.20–7.21 Static Testing

← Topic 7

Static testing examines software work products — code, design documents, requirements, test plans — without executing the program. It is a core verification activity that checks whether artefacts are built correctly according to specifications and standards. See §8.3 Verification & Validation for how static testing fits the broader V&V framework.

What is static testing? (7.20)

  • Static testing reviews code, design, and documentation to find defects without running the software.
  • It is a verification activity — it answers "Are we building the product right?" by checking conformance to plans and standards.
  • Static testing can begin early in the SDLC — requirements and design can be reviewed before any code is compiled or executed.
  • Defects found early are cheaper to fix than defects discovered during dynamic testing or in production.
  • Static testing complements dynamic testing; both are needed for comprehensive quality assurance.

Three static testing techniques (7.21)

1. Walkthrough

  • A walkthrough is an informal, author-led review where the author presents the work product and explains how it meets requirements.
  • Participants ask questions and suggest improvements — the goal is learning and early defect detection, not formal sign-off.
  • No strict checklist is required; the atmosphere is collaborative rather than adversarial.
  • Walkthroughs are commonly used for requirements, design, and code at early stages.

2. Code inspection

  • Code inspection is a formal peer review of source code conducted by reviewers other than the author.
  • Reviewers follow a checklist covering logic errors, standards compliance, documentation, and common defect patterns.
  • Each defect is logged with severity and location; the author fixes issues and a follow-up inspection may verify corrections.
  • Inspections are more rigorous than walkthroughs and produce measurable defect counts for process improvement.

3. Formal Technical Review (FTR)

  • FTR is a structured meeting with a defined agenda, roles, and deliverables — more formal than a walkthrough.
  • A trained moderator (not the author) leads the session, keeps discussion focused, and ensures all checklist items are covered.
  • Participants include the author, reviewers, and sometimes a customer representative or domain expert.
  • FTR produces an issue log, summary report, and action items assigned to responsible persons with deadlines.
  • Issue identification covers defects, deviations from standards, and areas needing clarification — not just syntax errors.

Compliance with design and coding standards

  • Static reviews check adherence to IEEE coding standards, project-specific naming conventions, indentation rules, and documentation requirements.
  • Reviewers verify that code structure matches the approved design and that comments accurately describe module behaviour.
  • Standards compliance reduces maintenance cost and improves readability — especially important when multiple developers contribute to the same codebase.
  • Non-compliance items are recorded in the issue log and tracked to closure like functional defects.

DimensionStatic testingDynamic testing
ExecutionWork products examined without running the programSoftware is executed with test inputs
Activity typeVerification — conformance to specs and standardsValidation — behaviour against requirements in operation

ftr_process.pseudo 
PROCEDURE formal_technical_review(work_product):

  // Step 1 — Preparation
  moderator = assign_trained_moderator()          // NOT the author
  reviewers = select_peer_reviewers(work_product)
  distribute work_product + checklist + standards TO reviewers
  reviewers PREPARE individually BEFORE meeting

  // Step 2 — Conduct structured meeting
  moderator OPENS meeting WITH defined agenda
  author PRESENTS work_product briefly
  reviewers DISCUSS findings AGAINST checklist
  moderator RECORDS each issue IN issue_log:
    issue_id, description, severity, location, owner

  // Step 3 — Issue identification and classification
  FOR EACH finding IN issue_log DO
    CLASSIFY AS defect | standards_violation | clarification_needed
    ASSIGN action_item TO responsible_person WITH deadline
  END FOR

  // Step 4 — Deliverables
  GENERATE summary_report:
    work_product_reviewed, participants, issues_found,
    standards_compliance_status, overall_recommendation
  DISTRIBUTE summary_report + action_items TO team

  // Step 5 — Follow-up
  author FIXES recorded issues
  IF critical_defects_remain THEN
    SCHEDULE follow_up_review
  ELSE
    CLOSE review WITH sign_off FROM moderator
  END IF

END PROCEDURE
Exam question — Static Testing

Q: Define static testing. Describe walkthrough, code inspection, and FTR. Explain standards compliance checking. Compare static and dynamic testing.

Model answer points:

  • Static testing = review code/docs WITHOUT executing program; verification activity.
  • Walkthrough = author-led, informal; Code inspection = formal peer review with checklist; FTR = structured meeting, trained moderator, issue log + summary report + action items.
  • Standards: IEEE coding standards, naming, indentation, documentation checked during reviews.
  • Static = no execution, early, verification; Dynamic = executed, later, validation.
  • FTR moderator must NOT be the author; produces issue identification, summary report, and action items.
  • Common trap: static testing is NOT the same as static analysis tools alone — it includes human reviews (walkthrough, inspection, FTR).
