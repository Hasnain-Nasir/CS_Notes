Key takeaways — exam revision

← Topic 7
  • 7.1 What is testing: Process to find defects, verify requirements, reduce risk. Not proof of correctness — shows presence of defects, not absence.
  • 7.2 Testing vs QA vs QC: Testing = subset of QA (find defects); QA = process-focused, entire SDLC (§8.1); QC = product verification. Testing overseen by SQA (§8.2).
  • 7.3 STLC phases: Requirement analysis → Test planning → Test case development → Environment setup → Execution → Closure. Parallel to SDLC testing phase (§2.1).
  • 7.4 Seven principles: (1) Shows presence not absence of defects; (2) Exhaustive testing impossible; (3) Early testing; (4) Defect clustering/Pareto 80/20; (5) Pesticide paradox; (6) Context dependent; (7) Absence of errors fallacy.
  • Testing guidelines: Clear goals, comprehensive cases, test data, automate where appropriate, multiple techniques, TDD, document, continuous monitor, involve users, use testing framework.
  • Testing objectives: Find defects, build confidence, provide decision info, prevent defects, verify requirements met.
  • 7.5 Unit testing: Individual components/functions; developer performs; white box; first level; stubs/drivers not needed at pure unit level.
  • 7.6 Integration testing: Interface between modules; after unit; black box at integration level; white box when testing internal module interactions.
  • 7.7 System testing: Complete integrated system; functional + non-functional; real environment; entire product validated as whole.
  • 7.8 Acceptance testing: Validate against user requirements; UAT; alpha/beta are acceptance types; validation not just verification.
  • 7.9 Regression testing: Re-test after changes; ensure old functionality not broken; automated suites common; mandatory after corrective maintenance (§10.4).
  • 7.10 Testing order: Unit → Integration → System → Acceptance. Each level builds on previous; defects cheaper to fix at lower levels.
  • 7.11 Functional vs performance: Functional = what system does; Performance = how well under load (non-functional subset).
  • 7.12 Non-functional testing: Performance, usability, security, reliability, scalability — how well, not what.
  • 7.13 Top-down integration: Test high-level first; stubs simulate lower modules; structure-oriented; big-to-small; early design defect detection.
  • 7.14 Bottom-up integration: Test low-level first; drivers simulate main module; OOP-friendly; small-to-big; less stub overhead; needs test driver.
  • Stubs vs drivers: Stub = simulate submodule (top-down); Driver = simulate calling module, pass data to submodules (bottom-up).
  • 7.15 White box testing: Tests internal structure/code; glass box/structural/clear box; developer performs. Techniques: statement, branch, path, condition coverage, basis path testing. Link cyclomatic complexity V(G) = basis paths (§6.6). Workflow: understand code → identify paths (CFG) → design cases → execute & measure coverage → report. Data flow + control flow testing.
  • 7.16 Black box techniques: Equivalence partitioning, boundary value analysis, decision table, state transition, use case testing, error guessing. No internal code knowledge; specification-based.
  • 7.17 Test data preparation: Valid/invalid/boundary classes; representative from each equivalence class; decision table → test cases; cause-effect graphing → decision table → test cases. Workflow: input → SUT → actual output → compare expected → pass/fail. Types: functional, regression, non-functional.
  • 7.18 Alpha testing: Internal acceptance at developer site; QA + internal users; controlled environment; before beta; bugs fixed before release.
  • 7.19 Beta testing: External users in real environment; limited users; feedback collection. Closed beta = invited only; Open beta = any user. Both validation/acceptance testing.
  • Alpha vs beta: Alpha: internal, developer site, controlled, before beta. Beta: external, user site, real-world, after alpha.
  • 7.20 Static testing: Review code/docs WITHOUT executing program; verification activity; early, cheaper fixes. Complements dynamic testing (§8.3).
  • 7.21 Static techniques: Walkthrough (author-led, informal); Code inspection (formal peer review, checklist); FTR (structured meeting, trained moderator NOT author, issue log + summary report + action items). Standards: IEEE coding, naming, indentation, documentation.
  • Static vs dynamic: Static = no execution, verification, early; Dynamic = executed, validation, after build.
  • 7.22 Manual vs automation: Manual = slower, human judgment, exploratory/usability/ad-hoc. Automation = faster, repeatable, higher setup cost, script maintenance; best for regression, repetitive tests, large data, stable UI.
  • 7.23 Selenium: Open-source web automation; Java/Python/C#; WebDriver API; cross-browser; TestNG/JUnit. Components: IDE (record), WebDriver (programmatic), Grid (distributed parallel).
  • 7.24 Smoke testing: Shallow wide; main functions after build; build verification; "does it smoke?"; reject bad builds before deeper testing.
  • 7.25 Sanity testing: Narrow deep; specific changed functionality after minor fix; follows smoke pass; rejects bad fixes.
  • Smoke vs sanity: Smoke = wide/shallow/new build; Sanity = narrow/deep/specific change.
  • 7.26 Load testing: Normal/expected load; performance under anticipated user volume; response time, throughput at typical scale.
  • Stress testing: Beyond normal capacity; breaking point; failure and recovery behaviour; capacity planning.
  • Load vs stress: Load = expected volume; Stress = exceed limits to find failure point.
  • Frontend testing: UI, UX, client-side logic, browser compatibility, responsiveness.
  • Backend testing: APIs, database, server logic, integration layers, data integrity.
  • 7.27 Testing vs debugging: Testing finds defects (tester, planned); Debugging locates/fixes root cause (developer, reactive to failures).
  • 7.28 Debugging process: Reproduce → narrow location → analyse cause → fix → retest → regression. Techniques: print/logging, breakpoints, binary search in code, rubber duck, cause elimination.
  • Exam traps: Alpha ≠ beta (internal/controlled vs external/real). Smoke ≠ sanity (wide build vs narrow fix). Load ≠ stress (expected vs beyond capacity). Static ≠ dynamic (no execution vs run program). Testing ≠ debugging (find vs fix). Top-down uses stubs; bottom-up uses drivers. White box = structural/developer; black box = functional/specification. Verification (static/right product built) vs validation (dynamic/right product) — §8.3.
