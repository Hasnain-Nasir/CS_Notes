7.27–7.28 Debugging

← Topic 7

Testing and debugging are complementary but distinct activities. Testing discovers that a defect exists; debugging locates its root cause and removes it. After a fix, retesting and regression testing confirm the correction — mandatory after corrective maintenance as described in §10.4 Maintenance Categories.

What is debugging?

  • Debugging is the process of finding and removing software errors (bugs) from a program.
  • It typically follows a failed test or user bug report — the tester provides symptoms; the developer traces the cause.
  • Debugging requires understanding both the expected behaviour (from specifications) and the actual faulty behaviour (from test output).
  • A fix is not complete until the original test passes and regression tests confirm no new defects were introduced.

Debugging steps

  1. Reproduce the bug — run the failing test case or follow the user steps until the defect appears consistently.
  2. Narrow the location — isolate which module, function, or line of code is responsible using logs, breakpoints, or binary search.
  3. Analyse the cause — determine why the code produces wrong output — logic error, off-by-one, null reference, race condition, etc.
  4. Fix the defect — modify the code to correct the root cause, not just the symptom.
  5. Retest — rerun the failing test case to confirm the fix resolves the reported issue.
  6. Regression test — run the broader test suite to ensure the fix did not break existing functionality — critical after corrective maintenance (§10.4).

Debugging techniques

  • Print / logging: Insert output statements or use a logging framework to trace variable values and execution flow at runtime.
  • Breakpoints: Use a debugger to pause execution at specific lines and inspect variable state, call stack, and memory.
  • Binary search in code: Comment out or disable half the code, rerun the test, and repeat — systematically halving the search space to locate the fault.
  • Rubber duck debugging: Explain the code line by line to another person (or an inanimate object) — verbalising often reveals the logical error.
  • Cause elimination: List possible causes, test each hypothesis systematically, and eliminate those that do not match the observed symptoms.
debug_process.pseudo 
PROCEDURE debug_defect(failed_test):

  // Step 1 — Reproduce
  result = RUN failed_test
  IF result.status != "FAIL" THEN
    RETURN "cannot_reproduce — gather more info from tester"
  END IF
  LOG symptoms: input, expected_output, actual_output

  // Step 2 — Narrow location
  suspect_modules = isolate_using_logs_and_breakpoints(failed_test)
  WHILE suspect_modules.size > 1 DO
    // Binary search: disable half, rerun
    half = suspect_modules.split_in_half()
    IF RUN failed_test WITH half[0] disabled STILL FAILS THEN
      suspect_modules = half[0]
    ELSE
      suspect_modules = half[1]
    END IF
  END WHILE
  fault_location = suspect_modules[0]

  // Step 3 — Analyse cause
  root_cause = analyse_code_at(fault_location)
  // e.g. off_by_one, null_pointer, wrong_operator, race_condition

  // Step 4 — Fix
  APPLY code_fix AT fault_location TO address root_cause

  // Step 5 — Retest
  retest_result = RUN failed_test
  IF retest_result.status != "PASS" THEN
    GOTO Step 2   // fix incomplete — continue debugging
  END IF

  // Step 6 — Regression
  regression_result = RUN full_regression_suite()
  IF regression_result.has_new_failures THEN
    REVERT fix OR debug_regression_failures
  ELSE
    CLOSE defect WITH root_cause_documentation
  END IF

END PROCEDURE
Exam question — Debugging

Q: Distinguish testing from debugging. Describe the debugging process and common techniques. Explain the role of regression testing after a fix.

Model answer points:

  • Testing finds defects (tester); debugging locates and fixes root cause (developer).
  • Testing can be planned; debugging is reactive to test failures.
  • Steps: reproduce → narrow location → analyse cause → fix → retest → regression.
  • Techniques: print/logging, breakpoints, binary search in code, rubber duck, cause elimination.
  • Regression testing mandatory after fix — especially corrective maintenance (§10.4).
  • Common trap: fixing symptoms without root cause analysis leads to recurring defects.
