7.22–7.23 Automation Testing

← Topic 7

Test automation uses scripts and tools to execute predefined test cases without manual intervention. It is especially valuable for regression testing after changes — see §7.9 Regression Testing. Automation does not replace manual testing entirely; the choice depends on test type, stability of the application, and cost of script maintenance.

Manual vs automation testing (7.22)

DimensionManual testingAutomation testing
SpeedSlower — human executes each stepFaster — scripts run repeatedly without fatigue
Initial costLower setup cost — no tool developmentHigher upfront cost — tool selection, script development, training
RepeatabilityVariable — human error and inconsistencyHigh — same steps executed identically every run
Human judgmentStrong — exploratory, usability, subjective assessmentLimited — checks only what is scripted
MaintenanceNo scripts to update when UI changesScripts must be maintained when application changes — ongoing cost
Best suited forExploratory, usability, ad-hoc, one-time testsRegression, repetitive suites, large data sets, stable functionality

When to automate

  • Automate repetitive tests that run frequently — login flows, CRUD operations, smoke suites after every build.
  • Automate regression testing to verify that fixes and enhancements do not break existing functionality — essential after maintenance changes (see §10.4).
  • Automate tests involving large data sets where manual entry is impractical or error-prone.
  • Automate when the UI and APIs are stable — frequent UI redesigns increase script maintenance cost and reduce automation ROI.
  • Automate performance and load test scenarios that require many concurrent virtual users.

When to keep manual testing

  • Use manual testing for exploratory testing — testers creatively probe the application without predefined scripts.
  • Use manual testing for usability and UX evaluation where human perception and judgment are essential.
  • Use manual testing for one-time or ad-hoc tests that will not be repeated enough to justify automation cost.
  • Use manual testing when requirements are volatile and scripts would be rewritten after every sprint.
  • Use manual testing for new features before they stabilise enough to automate reliably.

Selenium

  • Selenium is an open-source framework for automating web browser interactions — the most widely used web UI automation tool.
  • It supports multiple programming languages: Java, Python, C#, and others through language bindings.
  • Selenium uses the WebDriver API to control browsers programmatically — click buttons, fill forms, verify page content.
  • It provides cross-browser support — Chrome, Firefox, Edge, Safari — enabling consistent tests across environments.
  • Selenium integrates with test frameworks such as TestNG and JUnit for test organisation, assertions, and reporting.

Selenium components

  • Selenium IDE: Browser extension for record-and-playback of user actions — useful for quick prototyping of test scripts, not for large production suites.
  • Selenium WebDriver: Programmatic API for writing robust, maintainable test scripts in code — the core component for professional automation.
  • Selenium Grid: Distributes test execution across multiple machines and browsers in parallel — reduces total test run time in CI/CD pipelines.
selenium_test_example.pseudo 
TEST login_with_valid_credentials:

  // Setup — launch browser via WebDriver
  driver = WebDriver.create("chrome")
  driver.navigate_to("https://app.example.com/login")

  // Step 1 — Enter username
  username_field = driver.find_element(id="username")
  username_field.type("testuser@example.com")

  // Step 2 — Enter password
  password_field = driver.find_element(id="password")
  password_field.type("SecurePass123")

  // Step 3 — Click login button
  login_button = driver.find_element(id="btn-login")
  login_button.click()

  // Step 4 — Verify successful login
  welcome_message = driver.find_element(class="welcome-banner")
  ASSERT welcome_message.text CONTAINS "Welcome, testuser"
  ASSERT driver.current_url == "https://app.example.com/dashboard"

  // Teardown
  driver.quit()

END TEST
Exam question — Automation Testing

Q: Compare manual and automation testing. When should tests be automated vs manual? Describe Selenium and its components.

Model answer points:

  • Manual = slower, human judgment, exploratory/usability; Automation = faster, repeatable, best for regression.
  • Automate: repetitive, regression, large data, stable UI. Manual: exploratory, usability, one-time, ad-hoc.
  • Selenium = open-source web automation; Java/Python/C#; WebDriver API; cross-browser; TestNG/JUnit.
  • Components: IDE (record), WebDriver (programmatic), Grid (distributed parallel execution).
  • Automation scripts require maintenance when UI changes — not a one-time investment.
  • Common trap: automation cannot fully replace manual testing — exploratory and usability need humans.
