5.10 Coupling and Cohesion (+ Measures)

← Topic 5

Coupling and cohesion are the two primary parameters for measuring the quality of modularity in a software design. Together they determine maintainability, scalability, and reliability of the system.

  • Coupling measures the degree of interdependence between modules — an inter-module concept.
  • Cohesion measures the functional strength of elements within a module — an intra-module concept.
  • Design goal: Low coupling and high cohesion — modules should be independent of each other but strongly focused internally.

Coupling

Coupling is the measure of the degree of interdependence between modules. A good software design will have low coupling.

Types of coupling (worst → best)

  • 1. Content coupling (worst): One module is a part of or directly modifies the internal code/data of another module. Example: Module A directly jumps into the middle of Module B's code. Most undesirable — changes in one module force changes in another.
  • 2. Common coupling: Two modules share global data. Example: Both modules read/write a shared global variable; changes to the variable affect both modules and can cause synchronisation issues between processes.
  • 3. External coupling: Two modules depend on externally imposed data formats or devices outside the software logic. Example: Both modules depend on a specific data bus, CPU register, or hardware protocol.
  • 4. Control coupling: One module controls the flow of another by passing control information (flags, mode switches). Example: Module A passes a flag to Module B telling it which sub-function to execute — changes to A's control logic affect B.
  • 5. Stamp coupling: Two modules communicate by passing a data structure (record, array) when only part of the structure is needed. Example: Passing an entire student record when only the marks field is required.
  • 6. Data coupling (best): Two modules communicate only by passing simple data values (parameters). Example: Addition using call-by-value — only the numeric values are passed, with no shared state or control flags. Most desired type of coupling.

Cohesion

Cohesion is the measure of the functional strength of a module — how closely related its elements are. A good software design will have high cohesion.

Types of cohesion (worst → best)

  • 1. Coincidental cohesion (worst): The only relationship between functions in a module is that they happen to be in the same file — random or no logical connection. Example: A module containing unrelated utility functions grouped arbitrarily.
  • 2. Logical cohesion: All elements perform similar or slightly similar operations but serve different purposes. Example: A module containing mouse, printer, and scanner functions because they are all "device" operations.
  • 3. Temporal cohesion: Functions are related because they must all be executed in the same time span, with flow control between them. Example: A module with one function that catches exceptions and another that displays an error message — both run during error handling.
  • 4. Procedural cohesion: Functions are related through a flow of control — they are steps in an algorithm or procedure. Example: One function checks file access permission and another opens the file — sequential steps in a process.
  • 5. Communicational cohesion: Different functions operate on the same data structure. Example: Push and pop functions in a stack module — both operate on the same stack data structure.
  • 6. Sequential cohesion: Functions form a sequence where the output of one function is the input of the next. Example: Function1 reads raw data and Function2 processes that data — data flows sequentially through the module.
  • 7. Functional cohesion (best): All functions in a module cooperate to perform a single, well-defined task. Example: A mouse-control module with functions for reading trackball position, decoding movement, and updating the cursor — all parts of one function: controlling the mouse.

Impact on software quality

  • High coupling + low cohesion: System is difficult to change, test, and maintain — a change in one module can break many others.
  • Low coupling + high cohesion: System is easier to maintain, improve, and scale — changes are localised.
  • Both coupling and cohesion are important factors in determining maintainability, scalability, and reliability.

Coupling and cohesion at architectural level

  • During architectural design, designers do not stress individual module internals but concentrate on coupling and cohesion between modules.
  • Modules are treated as black boxes; the focus is on how they communicate and depend on each other.

Exam question — Coupling and cohesion

Q: Define coupling and cohesion. List all types with examples. What is the design goal?

Model answer points:

  • Coupling = inter-module dependence (low is good); cohesion = intra-module strength (high is good).
  • Coupling types (6): content → common → external → control → stamp → data (best).
  • Cohesion types (7): coincidental → logical → temporal → procedural → communicational → sequential → functional (best).
  • Goal: high cohesion + low coupling.
  • Give one example for each type (e.g., data coupling = call-by-value; functional cohesion = mouse control module).
