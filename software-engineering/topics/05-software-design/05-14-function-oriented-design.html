5.14 Function-Oriented Design

← Topic 5

Function-oriented design (FOD) is an approach to software design where the system is decomposed into a set of interacting units, each with a clearly defined function. The focus is on what the program does — the functions it performs — rather than the data it manipulates.

FOD uses a top-down strategy (§5.13) and is carried out using structured analysis and structured design techniques.

Two levels of the design process

  • Level 1: Decide which modules are needed based on the SRS and how modules should be interconnected.
  • Level 2: Design the internal logic of each module — algorithms, data structures, control flow.

Stepwise refinement

  • Function-oriented design is based on stepwise refinement — a top-down strategy where a program is refined as a hierarchy of increasing levels of detail.
  • Start with a high-level description of what the software/program does.
  • Refine each part by specifying in greater detail the functionality of each part.
  • Continue until each module's logic is at the statement level of the programming language.
  • Refinement is a process of elaboration — from a highly conceptual model to lower-level details.

Key tools in function-oriented design

1. Data Flow Diagram (DFD)

  • A DFD maps out the flow of information for any process or system — showing inputs, outputs, processes, data stores, and external entities.
  • DFDs are created during requirements analysis and refined during design.
  • Full coverage in §4.11.

2. Structure chart

  • For function-oriented design, the design is represented graphically by structure charts.
  • The structure of a program is made up of modules together with interconnections between modules.
  • Structure chart is a nice representation for a design that uses functional abstraction.
  • Full coverage in §5.7.

Module categories in FOD

  • Input module: Obtains data from external sources or subordinates and passes it upward to the calling module.
  • Output module: Sends processed data to external destinations or to superordinate modules.
  • Transform module: Performs computation or data transformation on input data.
  • Edit module: Validates, formats, or modifies data before further processing.
  • Coordinate module: Primary concern is managing the flow of data to and from different subordinates.
  • Get/put modules: Some modules obtain information from subordinates and pass it to superordinates (get); others receive from superordinates and pass to subordinates (put).

FOD vs object-oriented design

  • FOD focuses on functions; OOD focuses on data/objects — they are orthogonal approaches.
  • FOD uses top-down decomposition; OOD uses bottom-up composition.
  • FOD tools: DFD, structure charts; OOD tools: UML diagrams.
  • Full comparison in §5.13 and §5.15.
Exam question — Function-oriented design

Q: Explain function-oriented design. What tools are used? Describe stepwise refinement.

Model answer points:

  • Define FOD: decompose into interacting functional units; focus on what program does.
  • Top-down stepwise refinement: high-level description → refine to statement level.
  • Tools: DFD (analysis), structure chart (design representation).
  • Module categories: input, output, transform, edit, coordinate.
  • vs OOD: functions vs objects; top-down vs bottom-up.
