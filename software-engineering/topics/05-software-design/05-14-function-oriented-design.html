5.14 Function-Oriented Design

← Topic 5

The design process for software systems often has two levels. At the first level, the focus is on deciding which modules are needed for the system based on the SRS (Software Requirements Specification) and how the modules should be interconnected. At the second level, the internal logic of each module is designed.

Function-oriented design (FOD) is an approach to software design where the design is decomposed into a set of interacting units, each with a clearly defined function. The focus is on what the program does — the functions it performs — rather than the data it manipulates.

FOD uses a top-down strategy (§5.13) and is carried out using structured analysis and structured design techniques.

Generic procedure — stepwise refinement

Function-oriented design follows a generic top-down procedure:

  • Start with a high-level description of what the software or program does.
  • Refine each part of the description by specifying in greater detail the functionality of each part.
  • Continue refining until each module's logic reaches the statement level of the programming language.
  • This leads to a top-down structure — also called stepwise refinement.
  • Refinement is a process of elaboration from a highly conceptual model to lower-level details.

Problem in top-down design — single parent modules

In a strict top-down design, each module is typically used by at most one other module, and that module is called its parent module. This creates a rigid hierarchy where a module cannot easily be shared across different parts of the system.

  • If two unrelated parts of the system need the same logic, it may be duplicated in separate modules.
  • This increases redundancy and makes maintenance harder — a fix in one copy may be missed in another.

Solution — reusable modules

The solution is to design reusable modules — modules that can be used by several other modules to perform their required functions.

  • A reusable module is invokable from any part of the system that needs its functionality.
  • In structure charts, reusable modules are drawn as library modules with double vertical lines on the sides — see §5.7.
  • Example: a ValidateInput module used by both the Admission module and the Fee Payment module in a university system.

Function-oriented design strategies

Function-oriented design uses a set of structured tools that work together from analysis through detailed design. The four main strategies are:

1. Data Flow Diagram (DFD)

  • A DFD maps out the flow of information for any process or system.
  • It uses defined symbols — rectangles, circles, and arrows — plus short text labels to show data inputs, outputs, storage points, and the routes between each destination.
  • DFDs are created during requirements analysis and refined during design.
  • Full coverage in §4.11.

2. Data dictionary

  • A data dictionary is a repository that stores information about all data items defined in DFDs.
  • At the requirements stage, the data dictionary contains data items with their definitions.
  • Each entry typically includes: name of the item, aliases (other names), description/purpose, related data items, range of values, and data structure definition/form.
  • Data dictionaries eliminate ambiguity about what each data flow in a DFD actually represents.

3. Structure charts

  • A structure chart is the hierarchical representation of a system that partitions it into black boxes.
  • Components are read from top to bottom and left to right.
  • When a module calls another, it views the called module as a black box, passing required parameters and receiving results.
  • Full coverage in §5.7.

4. Pseudocode

  • Pseudocode is a system description in short English-like phrases describing the function of each module.
  • It uses keywords and indentation to represent control flow without language-specific syntax.
  • Pseudocode can replace flowcharts for documenting module logic and decreases the amount of documentation required.
  • Full coverage in §5.8.

Structure charts in function-oriented design

For function-oriented design, the design can be represented graphically by structure charts. The structure of a program is made up of the modules of that program together with the interconnections between modules. The structure chart of a program is a graphic representation of its structure.

  • In a structure chart, a module is represented by a box with the module name written inside it.
  • In general, procedural information is not represented in a structure chart, and the focus is on representing the hierarchy of modules.
  • However, there are situations where the designer may wish to communicate certain procedural information explicitly, such as major loops and decisions.
  • Such information can also be represented in a structure chart using conditional call and loop symbols.
  • A structure chart is a nice representation for a design that uses functional abstraction.

Module categories in FOD

Modules in a function-oriented system can be categorised into the following classes:

  • Input module: Obtains information from external sources or subordinates and passes it upward to the calling module.
  • Output module: Takes information from its superordinate and passes it on to its subordinates or external destinations.
  • Transform module: Exists solely for transforming data into some other form — performs computation or data transformation.
  • Coordinate module: Primary concern is managing the flow of data to and from different subordinates.
  • Edit module: Validates, formats, or modifies data before further processing.

Example — payroll system (FOD pipeline)

The following example shows how function-oriented design tools work together for a payroll system:

Step 1 — DFD (analysis)

  • External entities: Employee, HR Department, Bank.
  • Processes: Calculate Salary, Generate Payslip, Transfer Payment.
  • Data stores: Employee Records, Salary History.
  • Data flows: Employee details flow into Calculate Salary; calculated salary flows to Generate Payslip and Transfer Payment.

Step 2 — Structure chart (design)

  • Top-level control module: Payroll System.
  • Sub-modules: Get Employee Data (input), Calculate Salary (transform), Generate Payslip (output), Transfer Payment (output).
  • Reusable library module: Validate Employee ID — used by both Get Employee Data and Calculate Salary.
  • Coordinate module: Payroll System manages data flow between all sub-modules.

Step 3 — Pseudocode (detailed design)

calculate_salary.pseudo 
MODULE CalculateSalary(employeeId)
  SET employee = GetEmployeeData(employeeId)
  SET basicPay = employee.basicSalary
  SET allowances = employee.hra + employee.da
  SET deductions = employee.tax + employee.pf
  SET netSalary = basicPay + allowances - deductions
  RETURN netSalary
END MODULE

Advantages of function-oriented design

  • Provides a clear, systematic top-down path from requirements to implementation.
  • DFDs and structure charts make the system easy to understand for both technical and non-technical stakeholders.
  • Modules with clearly defined functions are easier to test, debug, and maintain individually.
  • Reusable modules reduce redundancy and promote code reuse across the system.
  • Well suited for structured programming languages like C, COBOL, and Fortran.

Disadvantages of function-oriented design

  • Strict top-down hierarchy can make module reuse difficult without explicit library module design.
  • Changes to data structures can affect many modules because functions operate on shared data flows.
  • Not well suited for systems where data and behaviour are tightly bound to real-world entities.
  • Large systems can produce very complex DFDs and structure charts that are hard to manage.

When to use function-oriented design

  • When the system can be naturally described as a set of data transformations and processes.
  • When using structured or procedural programming languages.
  • When requirements are well understood and can be decomposed top-down — e.g., payroll, billing, or inventory systems.
  • When the design team is familiar with DFDs, structure charts, and stepwise refinement.

FOD vs object-oriented design

AspectFunction-Oriented DesignObject-Oriented Design
Focus What the program does — functions and processes. What entities exist — objects and their interactions.
Strategy Top-down decomposition (stepwise refinement). Bottom-up composition (identify objects, then integrate).
Tools DFD, data dictionary, structure chart, pseudocode. UML — class, sequence, use case diagrams.
Languages C, COBOL, Fortran (structured/procedural). C++, Java, Python (object-oriented).
Data handling Data flows between functions as parameters. Data encapsulated within objects.

Exam question — Function-oriented design

Q: Explain function-oriented design. What tools are used? Describe stepwise refinement and module categories.

Model answer points:

  • Define FOD: decompose into interacting functional units; focus on what program does; two-level design process.
  • Generic procedure: high-level description → refine each part → top-down structure (stepwise refinement).
  • Problem: single parent module; solution: reusable/library modules.
  • Four strategies: DFD, data dictionary, structure chart, pseudocode.
  • Structure chart: module = box; hierarchy focus; loops/decisions when needed; functional abstraction.
  • Module categories: input, output, transform, coordinate, edit.
  • Example: payroll system — DFD → structure chart → pseudocode → data dictionary.
  • vs OOD: functions vs objects; top-down vs bottom-up; DFD/SC vs UML.
