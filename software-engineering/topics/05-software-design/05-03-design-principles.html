5.3 Software Design Principles

← Topic 5

Good software design follows a set of guiding principles that make systems easier to manage, maintain, test, and extend. These principles apply at every level — from overall architecture down to individual modules — and are central to exam questions on design quality.

Key design principles

1. Modularity

  • Modularity means breaking software into smaller, independent sections or modules, each handling a specific task.
  • Each module can be developed, tested, and maintained separately without affecting the entire system.
  • See §5.6 for detailed coverage.

2. Low coupling

  • Coupling measures how much one part of the software depends on another.
  • The goal is to minimise connections between modules so that changing one module does not ripple through others.
  • Low coupling makes software more flexible and easier to update.
  • Full coupling types covered in §5.10.

3. High cohesion

  • Cohesion measures how closely related the elements within a module are — how well they work together for a single purpose.
  • High cohesion means each module does one well-defined job; low cohesion means unrelated tasks are grouped together.
  • Design goal: high cohesion, low coupling.

4. Abstraction

  • Abstraction simplifies software by hiding unnecessary complexity and showing only essential features.
  • It makes software easier to use and understand without overwhelming users or developers with implementation details.
  • At design time, abstraction lets designers focus on what a module does before deciding how it does it.

5. Anticipation of change

  • Software should be designed with future changes in mind — new features, new technologies, or evolving requirements.
  • Modular, loosely coupled designs adapt more easily than monolithic, tightly coupled ones.

6. Simplicity

  • Simple designs have fewer errors and are easier to maintain and improve over time.
  • Avoid over-engineering: include only what is needed to satisfy requirements, not unnecessary complexity.

7. Sufficiency and completeness

  • The design must meet all required functions without unnecessary extras.
  • Every SRS requirement must map to a design element; avoid gold-plating features not in the specification.

Design principles in practice

  • At architectural level, modularity and low coupling guide how major components are separated.
  • At module level, high cohesion ensures each unit has a clear, single responsibility.
  • At interface level, abstraction hides internal details behind clean, stable APIs.
  • These principles underpin both function-oriented design (§5.14) and object-oriented design (§5.15).
Exam question — Design principles

Q: List important software design principles. Why are high cohesion and low coupling desirable?

Model answer points:

  • List: modularity, low coupling, high cohesion, abstraction, anticipation of change, simplicity, sufficiency/completeness.
  • High cohesion: module elements work together for one purpose → easier to understand, test, reuse.
  • Low coupling: modules independent → changes isolated, system flexible.
  • State design goal: high cohesion + low coupling = good modular design.
