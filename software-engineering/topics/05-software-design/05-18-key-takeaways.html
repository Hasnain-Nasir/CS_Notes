Key takeaways — exam revision

← Topic 5
  • Software design: Transforms SRS → SDD; most creative SDLC phase; decides how requirements are implemented. Good SDD: complete w.r.t. SRS, readable, covers data/functional/behavioural domains.
  • Conceptual vs technical design: Conceptual = what system will do (for customer); technical = how builders implement it (hardware, modules, data structures).
  • Three design levels: Interface (black-box I/O, events, data formats, timing) → Architectural (major modules, responsibilities, coupling/cohesion) → Detailed/LLD (algorithms, data structures, unit internals).
  • Design process (Set 2): Modules, control relationships, interfaces, data structures, algorithms designed. Preliminary design → program architecture; detailed design → module specification document.
  • Design phases: Understand requirements → research/plan → design → technical design → UI design → prototyping.
  • Design elements: Architecture, modules, components, interfaces, data.
  • Design principles: Modularity, low coupling, high cohesion, abstraction, anticipation of change, simplicity, sufficiency/completeness. Goal: high cohesion + low coupling.
  • Architectural design: IEEE — components + connectors + constraints + semantic models. Styles: layered (4-tier: presentation/business/application/data), OO (message passing), client-server, pipe-filter, event-driven, microservices.
  • LLD vs HLD: HLD = module decomposition, black boxes; LLD = internal white-box detail, algorithms, data structures. Deliverable = module specification document. Techniques: pseudocode, flowcharts, structure charts, UML.
  • Modularization: Independent units + well-defined interfaces. Advantages: easy to understand, design/document, code/test, maintain.
  • Structure charts: Hierarchical module architecture; black-box modules. Symbols: module box, conditional call, loop (curved arrow), data flow (empty circle), control flow (filled circle), storage. Module types: control, sub, library. Categories: input, output, transform, edit, coordinate.
  • Pseudocode: Language-independent algorithm planning; mix of natural language + programming constructs; not executable; part of SDD documentation.
  • Flowcharts: Graphical control flow; symbols: terminal (oval), process (rectangle), decision (diamond), I/O (parallelogram), connector, arrows. vs structure chart: control flow vs architecture; no inter-module data.
  • Coupling (6 types, worst→best): Content (module modifies another's internals) → Common (shared global data) → External (hardware/external format) → Control (flags/mode switches) → Stamp (passes data structure) → Data (simple parameter passing — best).
  • Cohesion (7 types, worst→best): Coincidental (random grouping) → Logical (similar ops, different purposes) → Temporal (same time span) → Procedural (algorithm steps) → Communicational (same data structure) → Sequential (output→input chain) → Functional (single task — best).
  • Cohesion vs coupling: Cohesion = intra-module strength (want HIGH); coupling = inter-module dependence (want LOW). Cohesion = functional strength; coupling = independence. Best: functional cohesion + data coupling.
  • Design strategies: Top-down (stepwise refinement, small/medium, FOD), bottom-up (build components first, large, OOD), iterative, incremental, agile.
  • Top-down advantages: Systematic, modular, predictable cost/time. Disadvantage: very large/complex problems. Bottom-up advantages: Large projects, reusable components. Disadvantage: complex architecture.
  • Function-oriented design: Top-down; focus on functions; tools = DFD + structure charts; stepwise refinement to statement level; module categories: input, output, transform, edit, coordinate.
  • Object-oriented design: Bottom-up; focus on data/objects; four pillars: encapsulation, abstraction, inheritance, polymorphism; tools = UML; objects = independent entities with state + behaviour.
  • FOD vs OOD: Functions vs data focus; top-down vs bottom-up; DFD/structure charts vs UML; procedure-level vs class-level decomposition.
  • UML: Standard visual language for OOD. Structural: class (attributes, methods, associations, inheritance, multiplicity), component, deployment, package. Behavioural: use case (actors, boundary, include/extend), sequence (lifelines, messages, activation), activity (workflow, decisions, fork/join), state.
  • Use case diagram: Actors, use cases (ovals), system boundary (rectangle), association, include (dashed), extend (dashed). High-level functionality view.
  • Class diagram: Class = name + attributes + methods; relationships: association, inheritance (hollow arrow), aggregation (hollow diamond), composition (filled diamond); multiplicity (1, *, 1..n).
  • Sequence diagram: Lifelines, activation bars, messages (solid=sync, dashed=async); shows interaction order over time.
  • Activity diagram: Activities, decision/merge nodes, fork/join bars, initial/final nodes; workflow and use case steps.
  • UI design: Front-end view for user interaction. Good UI: attractive, simple, responsive, clear, consistent. Types: CLI (command prompt, remember syntax) vs GUI (visual, windows/icons/menus).
  • Interface level (black box): Incoming/outgoing events, data formats, ordering/timing; system internals ignored.
  • UI design process (4 stages, iterative/spiral): (1) User/task/environment analysis — user profiles, task identification, physical environment questions; (2) Interface design — interface objects, control mechanisms, user scenarios, response time, error handling, Mandel's rules; (3) Construction — prototype, UI toolkit (windows, menus, commands); (4) Validation — correct tasks, variety of tasks, easy to use/learn, user acceptance.
  • Theo Mandel's 3 golden rules: (1) Place user in control — no forced actions, flexible interaction, interruptible/undoable, customisable for experts, hide internals, direct object manipulation; (2) Reduce memory load — less short-term memory demand, meaningful defaults, intuitive mnemonics/shortcuts, real-world metaphors, progressive disclosure; (3) Make consistent — meaningful context/navigation indicators, consistency across app family, don't break established conventions.
  • UI design principles: User-centred design, consistency, simplicity, feedback (visual/message/sound), accessibility (contrast, font, screen readers), flexibility/customisation.
  • UI patterns: Navigation, forms, modals, cards, search/filter. Pattern model: problem → context → solution → examples.
