1.4 Software Components

← Topic 1

A program becomes software only when it is accompanied by proper documentation and operating procedures. In Software Engineering, we do not treat code alone as a finished product because other people must be able to understand, use, maintain, and audit the system after it is delivered.

Software has three essential components:

  1. Program: The program is the executable code (source and/or compiled) that contains the core logic and performs the required tasks. A program alone is not complete software because it cannot be properly maintained or used without supporting material.
  2. Documents: Documents include all descriptions, diagrams, and instructions related to the software's design, coding, testing, and preparation. This covers requirements specifications, design documents, test plans, and release notes. Documentation makes software understandable, maintainable, and auditable for developers and managers.
  3. Operating procedure (user and operational manuals): Operating procedures are guides that explain what the software does, how to install it, how to operate it, and how to control its activities. Without these manuals, end users and operators cannot use the product effectively in day-to-day work.

Why all three components matter

Each component serves a different audience. The program is mainly for the machine and the development team who write and fix the code. The documents help developers, testers, and future maintainers understand how the system was designed and verified. The operating procedures help non-technical users install and run the software correctly. If any one of these is missing, the product is incomplete and difficult to sustain over time.

Types of documentation (overview)

  • Requirements documentation: This describes what the software must do and forms the basis for design, testing, and acceptance.
  • Design documentation: This explains the system architecture, modules, and data structures so that developers can implement and later modify the system.
  • Testing documentation: This records test cases, test results, and defects found so that quality can be verified and traced.
  • User and operational manuals: These explain installation steps, daily operations, and troubleshooting for people who use the software but do not write code.

Example — Payroll application

A payroll application is delivered as complete software only when all three components are present. The program is the compiled application that calculates salaries and generates payslips. The documents include the SRS and design document that explain how salary rules, deductions, and reports are implemented for future maintainers. The operating procedure is the user manual that tells HR staff how to enter employee data, run monthly payroll, and print or export payslips.

Exam question — Software Components

Q: List and explain the three components of software. Why is documentation essential?

Model answer points:

  • The three components of software are the program, documents, and operating procedure or user manuals.
  • A program alone is not software because documentation and manuals are needed to complete the product for real use.
  • Documents support maintenance, testing, project handover, and compliance with organisational or legal requirements.
  • User and operational manuals enable correct installation and day-to-day use by people who are not developers.
  • Without proper documentation, a system becomes difficult to update when requirements or staff change.
