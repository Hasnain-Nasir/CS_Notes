1.9 Software Engineering Processes

← Topic 1

A software process (software methodology) is a set of related activities that leads to the production of software — either from scratch or by modifying an existing system. Without a defined process, team members may perform activities in random order, progress cannot be monitored, and the project will likely fail.

Note: Specific SDLC models (Waterfall, Spiral, Agile, etc.) are covered in Topic 2. Here we focus on the generic activities that all processes include.

Generic software process activities

  1. Feasibility study: The problem is defined in abstract terms, and financial and technical feasibility are checked through cost–benefit analysis, review of infrastructure and human resources, and comparison of alternative strategies.
  2. Requirements analysis and specification: Complete customer requirements are understood, project data is collected and analysed, and an SRS (Software Requirements Specification) is produced in natural language describing what the system must do, not how. Errors at this stage cause severe problems in later stages.
  3. Design: Requirements are transformed into a structure suitable for coding by choosing architecture and algorithms with attention to coupling and cohesion, and producing an SDD (Software Design Description).
  4. Coding: The design is translated into a programming language in a way that supports later testing and maintenance, with emphasis on readable and maintainable code.
  5. Testing: The program is executed to find bugs, because a bug that is executed becomes a failure, and human errors in development produce faults in the code.
  6. Implementation (deployment): The software is installed at the user site, users receive training, and hardware requirements are checked before operation begins.
  7. Maintenance: Any change made after official release is maintenance, which may be corrective to fix defects, adaptive to respond to environment changes, or perfective to improve performance and features.

Four framework activities

  • Software specification: This activity defines what is to be built and the constraints under which the system must operate.
  • Software development: This activity covers the design and programming work needed to create the system.
  • Software validation: This activity checks whether the finished product meets requirements and actual user needs.
  • Software evolution: This activity covers initial development and the later updates made to the system over time.
Exam question — Software Process

Q: Describe the main activities in a software engineering process. Why is requirements analysis critical?

Model answer points:

  • The answer should list all seven activities from feasibility study through to maintenance.
  • A defined software process provides structure, monitoring, and predictability for the project team.
  • Errors in the SRS propagate through later stages, so fixing requirements defects late is extremely costly.
  • Without a defined process, development becomes chaotic, which links back to the software crisis in §1.6.
